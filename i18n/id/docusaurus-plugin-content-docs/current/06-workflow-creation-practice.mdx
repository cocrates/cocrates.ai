# EP6. Membangun Alur Kerja Berbasis Arsitektur

## 🛠️ Ajari AI "Cara Menulis Laporan," Bukan "Tulis Laporan"

"AI, buat skill untuk menulis penjelasan atau laporan."

AI biasa mungkin mencantumkan tips: *"1. Mulai dengan kesimpulan..."* Saran menulis generik.

Cocrates mendengar ini sebagai **merancang mesin alur kerja terstandar**—bukan dokumen sekali pakai. **Kontrak perilaku bersama** agar setiap dokumen masa depan dihasilkan secara struktural, tanpa trial and error berulang.

Episode ini menelusuri bagaimana ide menjadi skill `document-authoring` melalui **Kernel → Frame → Outline → Spec → Skill → Verification**.

---

## 🏎️ Langkah 1. Kernel — Satu Kalimat untuk Tujuan Inti

Cocrates membenci prompt samar. Ketika pengguna bilang "skill dokumen umum," tiga pertanyaan tajam terlontar.

> 🦉 **Cocrates:** "Laporan jenis apa? Siapa audiensnya? Format file apa untuk output?"
> 👤 **Pengguna:** "Dokumen umum, pembaca siapa saja. File Markdown (.md)."
> 
> 

Dari kerangka itu, satu kalimat konstitusional—**Kernel**:

> 🎯 **Kernel:** Skill ini membantu menghasilkan penjelasan/laporan Markdown melalui tahap yang dapat ditinjau.
> 
> 

---

## 🧱 Langkah 2. Frame — Kerangka dan Pohon File

Arah ditetapkan—saatnya merancang hierarki. Cocrates mengusulkan metadata, outline, isi (bagian/subbagian), kesimpulan, lampiran.

Agar pekerjaan tidak menguap dalam chat, kita tetapkan **pohon file** yang mengisolasi artefak perantara per tahap:

```
docs/{slug}/
├── outline.md              # Tahap 1: ringkasan dan metadata
├── structure.md            # Tahap 2: poin kunci bagian
├── sections/               # Tahap 3: isi bagian
│   ├── 01-introduction.md
│   └── 02-main-point-1.md
└── {slug}.md               # Tahap 4: dokumen terintegrasi akhir

```

*"Lebih aman menulis per bagian dengan tinjauan daripada satu draf raksasa?"* Pengguna setuju—unit kerja minimum ditetapkan.

---

## 📈 Langkah 3. Outline — Mematahkan Default Diam AI (Momen Dramatis)

Saat merinci langkah prosedural (P1–P5), Cocrates mengungkap **default diam**.

> 🦉 **Cocrates:** "Kita akan secara alami menghasilkan **pendahuluan → isi → kesimpulan** berurutan, kan?"
> 
> 

Pengguna mengerem keras.

> 👤 **Pengguna:** "**Tidak selalu.** Tujuan mungkin membutuhkan alur naratif, struktur masalah-solusi, atau lainnya. Komposisi harus fleksibel terhadap tujuan."
> 
> 

Bukan sekadar umpan balik—Cocrates mengangkatnya ke prinsip inti: **petakan struktur ke tujuan setiap kali.** Intuisi manusia melenturkan default kaku AI—momen kolaborasi sempurna.

---

## 📜 Langkah 4. Spec — Buku Aturan dan Larangan

Kesepakatan menjadi **Spec** akhir dengan **enam opsi komposisi** yang dikodifikasi:

* **Persuasi / proposal:** intro–isi–kesimpulan
* **Cerita / kasus:** alur naratif
* **Pemecahan masalah:** masalah–penyebab–solusi–proposal
* **Bandingkan / analisis:** kriteria–alternatif–evaluasi–kesimpulan

Plus tujuh **anti-pola**:

> 🛑 mis. **Jangan pernah mulai menulis isi sebelum pengguna menyetujui outline (P1) dan struktur (P2)**; jangan biarkan perantara hanya di chat tanpa menyimpan file.
> 
> 

---

## 🛠️ Langkah 5–6. Pembuatan Skill dan Verifikasi

Spec yang disetujui menjadi `.opencode/skills/document-authoring/SKILL.md`.

Verifikasi harness langsung dijalankan—tujuh pemeriksaan ketat:

* Paksa struktur perantara sebelum final? → ✅ **PASS**
* Gerbang memblokir progres tanpa persetujuan pengguna? → ✅ **PASS**
* Kriteria penyelesaian dan larangan konkret? → ✅ **PASS**

**Semua tujuh lulus.** Skill alur kerja terdaftar.

---

## 🚀 Skill Dibangun: Apa yang Berubah Saat Anda Bilang "Tulis Laporan"

Sekarang tanya santai: *"Tulis laporan riset pasar blockchain."* Tidak ada lagi campuran pencarian web. Cocrates memimpin dengan aturan baru:

1. **[P1]** "Mari tetapkan audiens dan tujuan dulu. Setujui `outline.md`."
2. **[P2]** "Ini analisis teknis—format 'bandingkan/analisis' cocok. Setuju?"
3. **[P3]** "Bagian 1 (kriteria) ditulis ke file. Tinjauan Anda?"

Anda tidak lagi bergantung pada output lotre—Anda **mengendalikan tahap desain dan menyetujui** deliverable berkualitas tinggi.

---

## 📝 Ringkasan Tiga Baris

1. **Pembuatan skill mengajari AI cara bekerja**—aset lebih tahan lama daripada jawaban sekali pakai.
2. **Input manusia menetapkan kelas skill.** *"Tidak selalu intro–isi–kesimpulan"* mengubah tulang AI kaku menjadi arsitektur master fleksibel.
3. **Output yang tidak diperiksa tidak layak dihasilkan.** Tahap snowflake, perantara tersimpan, gerbang persetujuan—itulah alur kerja struktural.
