## Slide 6: 뼈대(Architecture)의 중요성
- **Section:** 본론 — Architecture
- **Purpose/goal:** 최종 산출물을 한 번에 찍어내지 않고, 단계별 구조 설계 → 검토 → 승인 → 다음 단계로 진행하는 과정을 블로그 예시로 보여준다.
- **Content:**
  - 블로그 시리즈 제작 4단계:
    - (1) 전체 그림(overview.md) 설계 → 검토/승인
    - (2) 각 에피소드 뼈대(outline.md) → 검토/승인
    - (3) 상세 설계(episodes.md) → 검토/승인
    - (4) 본문 생성 → 검토/승인
  - 매 단계마다 🔍 검토와 ✅ 승인이 들어간다
  - "이것이 Cocrates가 작동하는 방식이다"
  - 시각: 4단계 수직 흐름 다이어그램. 각 단계에 🔍 아이콘.
- **Type:** 사례 연구 / 프로세스 설명

## Slide 7: 검토의 진짜 본질 — U → A → E → A
- **Section:** 본론 — 검토의 본질
- **Purpose/goal:** 검토가 단순한 오타 확인이 아니라, 이해→분석→평가→승인의 4단계 과정임을 명확히 정의한다.
- **Content:**
  - "대부분의 사람들이 생각하는 검토: 대충 읽어보고 오타 확인"
  - Cocrates가 말하는 검토:
  - **① 이해 (Understand):** 내 말로 다시 설명할 수 있을 때 비로소 이해한 것
  - **② 분석 (Analyze):** "왜 이렇게 구현되었는가?" — 설계 의도, 구조, 로직 파악
  - **③ 평가 (Evaluate):** "이게 정말 최선인가?" — 프로젝트 목적/기준에 부합하는지 판단
  - **④ 승인 (Approve):** "내가 확인했다. 이걸로 간다. 책임은 나다."
  - 검토 = **U → A → E → A**
  - 시각: 4단계 흐름도 (U → A → E → A, 각 단계마다 아이콘과 설명)
- **Type:** 개념 정의 / 프로세스 설명

## Slide 8: 무지의 제거 (Harnessing Ignorance)
- **Section:** 본론 — 검토의 본질
- **Purpose/goal:** 검토의 궁극적 목적이 단순한 오류 수정이 아니라, 자신의 무지를 인식하고 채워가는 '성장'임을 전달한다.
- **Content:**
  - "검토의 목적은 단순한 오류 찾기가 아니다"
  - **무지의 제거 (Harnessing Ignorance)**
  - 내가 몰랐던 것을 알게 되는 과정 = 진짜 성장
  - Cocrates의 원칙: "do not leave ignorance unattended" — 무지를 방치하지 마라
  - 검토를 통해 모르는 것을 발견하고 채워 넣는 것
  - 시각: 빈 원(무지)에서 채워진 원(지식)으로 변화하는 아이콘. 또는 "?"에서 "!"로.
- **Type:** 개념 설명 / 핵심 메시지

## Slide 9: Cocrates Harness의 리듬
- **Section:** 본론 — Cocrates 역할
- **Purpose/goal:** Cocrates Harness가 어떻게 검토 없는 생성을 방지하는지 구체적인 대화 예시로 보여준다.
- **Content:**
  - 👤 **유저:** "귀찮아, 그냥 로그인 모듈 대충 만들어줘!"
  - 🦉 **Cocrates:** "검토되지 않은 산출물은 생성할 가치가 없습니다. 구조부터 함께 설계해볼까요?"
  - 처음에는 잔소리처럼 느껴질 수 있다
  - 하지만: **"뼈대를 검토하는 1시간이, 일주일짜리 버그 수정을 막아준다"**
  - Cocrates = 답을 주는 자판기가 아니라, 주도권을 지켜주는 **가드레일**
  - 시각: 대화 풍선 형식 (유저 vs Cocrates). 하단에 "가드레일" 아이콘.
- **Type:** 사례 연구 / 대화 예시
