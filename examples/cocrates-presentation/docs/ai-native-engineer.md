# AI-native Engineer — 정의와 개념

## 정의

**AI-native Engineer**란 AI에게 실제 업무를 수행하게 하고, 그 결과를 검토(이해 → 분석 → 평가 → 승인)하여 최종 산출물의 품질과 방향을 책임지는 사람을 말한다. AI가 일하고, 사람이 리드한다.

**AI-assisted Engineer**란 AI를 단순 업무를 보조하는 도구(계산기, 비서)로 사용하는 엔지니어를 말한다. 사람이 일하고, 필요한 경우에 AI를 사용한다.

핵심은 AI의 역할을 **무엇으로 보는가**에 있다.

---

## AI-assisted Engineer vs AI-native Engineer

이 둘은 **명확히 구분**된다. 스펙트럼이나 성장 단계의 문제가 아니라, AI를 대하는 **방식의 차이**다.

| 구분 | AI-assisted Engineer | AI-native Engineer |
|------|----------------------|-------------------|
| **AI의 역할** | 나를 **보조**하는 도구 | 나를 대신해 **일을 수행**하는 팀원 |
| **업무 방식** | 내가 일하고, AI가 도와줌 | AI가 일하고, 내가 검토 |
| **요청** | "로그인 모듈 짜줘" (지시) | "인증 방식별 장단점을 분석하고 추천해줘" (위임) |
| **검토** | 생략하거나 피상적 확인 | **이해 → 분석 → 평가 → 승인**의 4단계 검토 |
| **결과물 이해도** | 낮음 ("AI가 짰으니까 됐겠지") | 높음 ("이 구조가 선택된 이유를 설명할 수 있다") |
| **AI의 위치** | 도구 — 내 업무의 보조자 | 팀원 — 내 업무의 수행자 |
| **비유** | 계산기로 더하기만 하는 사람 | 팀원에게 업무를 위임하고 검토하는 팀장 |

### 핵심 차이: 일의 주체

```
AI-assisted:  [나] --(도움 요청)--> [AI(도구)] --(결과 수용)--> [나]
AI-native:    [나] --(지시/위임)--> [AI(팀원)] --(수행)--> [검토(U→A→E→A)] --> [나(승인)]
```

AI-assisted는 AI가 **나를 도와주는** 구조다. AI-native는 AI가 **일을 수행하고 내가 검토하는** 구조다. 이 차이는 절대적이다.

## 당신의 AI는 몇 명의 팀원인가?

AI-native Engineer라면, 다음 질문이 의미를 가진다.

> **"당신은 AI를 몇 명의 팀원으로 생각하는가?"**

AI-native 방식으로 일한다는 것은 AI를 팀원으로 활용한다는 뜻이다. 그렇다면:

- **1명?** — 1명 분량의 업무를 AI 팀원에게 위임하고 검토한다
- **4명?** — 4개 영역의 업무를 동시에 위임하고 검토한다
- **10명?** — 소규모 팀을 리드하듯 AI 팀원들을 운용한다
- **20명? 100명?** — (OpenAI 사례처럼) 500명 조직의 생산성을 3명이 낸다

이 질문은 **당신의 역량**을 묻는 질문이다. 몇 명의 AI 팀원을 효과적으로 리딩해서 업무를 수행할 수 있는가? 이것이 AI-native Engineer의 성장 방향이다.

```
AI-native Engineer의 스펙트럼 (명확한 구분 이후의 성장)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
AI-assisted와 AI-native는 명확히 구분된다.
                           │
AI-native Engineer가 된 후: "내 AI 팀원은 몇 명인가?"
                           │
                           1명 → 4명 → 10명 → 20명 → ... → N명
                           │
                           당신의 역량이 팀 규모를 결정한다
```

## 실제 사례: OpenAI의 Zero-Code Million-Line 프로젝트

### 이 사례가 중요한 이유

OpenAI의 사례는 **AI-native Engineer가 무엇인지**를 극명하게 보여준다. 이 엔지니어들은 직접 코딩하지 않았다. AI에게 일을 시키고, 그 결과를 검토했다. 그리고 100만 줄 이상의 프로덕션 코드를 출시했다.

### 개요

- **인력:** 3명의 엔지니어
- **기간:** 5개월
- **결과:** 100만 줄의 프로덕션 코드 — **단 한 줄도 직접 작성하지 않고**
- **생산성:** 전통적 개발 대비 **10배** 속도
- **산출물:** 약 1,500개의 Pull Request, 하루 최대 10억 토큰 처리

### 핵심: Symphony Harness

팀은 **Symphony**라는 오케스트레이션 시스템을 구축했다. 이는 AI 코딩 에이전트들을 자율적으로 운용하는 **Harness(하네스)** 다.

```
Ticket 생성 → Agent가 코드 작성 & PR 생성 → 자동 리뷰 & CI → Merge
                                                       ↓
                                        인간의 검토는 Post-Merge로 전환
```

인간은 더 이상 코드를 직접 작성하지 않는다. 대신 **에이전트를 가이드하는 Harness**를 설계하고, 에이전트가 생성한 결과를 검토한다.

### 인간의 새로운 역할: Architect & Gardener

| 전통적 역할 | AI 시대 역할 |
|------------|-------------|
| 코드를 직접 작성 | 시스템 병목을 식별하고 Harness로 해결책을 인코딩 |
| 코드 리뷰를 Pre-merge | Post-merge 검토로 전환 |
| 문서화는 부차적 | **Core Beliefs**, **Tech Tracker** 등 markdown 문서가 핵심 설계 도구 |
| 의존성 라이브러리 사용 | 중간 복잡도 의존성은 **내재화(Vendoring)** 하여 정밀 제어 |

Lopopolo는 이 역할을 **"group tech leading a 500 person organization"** 이라고 표현했다 — 3명이 500명 조직처럼 일하는 방식.

### 시사점

> **"If a business process can be expressed in code, a well-built harness can delegate it to AI."**
> (비즈니스 프로세스가 코드로 표현될 수 있다면, 잘 만들어진 Harness는 이를 AI에 위임할 수 있다.)

이 사례는 AI-native Engineer가 이미 현실에서 작동 중임을 증명한다. 직접 코딩하지 않고도 프로덕션 수준의 서비스를 출시할 수 있다. 그리고 그 핵심은 **Harness를 설계하고 검토하는 인간의 역량**에 달려 있다.

> 참고: https://finance.biggo.com/news/308f3eaef3245166 (2026.04.09, BigGo 편집팀)

## AI-native Engineer의 핵심 역량

### 1. 검토 (Examine) — 이해 → 분석 → 평가 → 승인

AI가 수행한 결과를 네 단계로 검토한다:

| 단계 | 질문 | 설명 |
|------|------|------|
| **이해 (Understand)** | "이 결과물이 왜 이렇게 구성되었는가?" | 구조와 의도를 파악. 내 말로 설명할 수 있어야 함. |
| **분석 (Analyze)** | "어떤 구조로 이루어져 있는가? 다른 대안과의 차이는?" | 구성 요소와 관계를 파헤치고 대안과 비교 |
| **평가 (Evaluate)** | "이것이 내 요구와 일치하는가? 품질 기준을 충족하는가?" | 기준과의 차이(Deviation)를 발견하고 가치를 평가 |
| **승인 (Approve)** | "이대로 최종 산출물로 확정하겠다." | 명시적 승인. 승인하지 않으면 수정/재요청 |

### 2. 구조적 설계 (Architectural Design)

AI에게 일을 위임하기 전에, 무엇을 어떤 순서로, 어떤 기준으로 수행할지 **구조를 먼저 설계**한다.

- 요청을 구체적인 단위로 분해
- 각 단위의 품질 기준과 성공 조건 정의
- AI의 출력물을 검증할 체크리스트 준비

### 3. 팀 리드 역량

더 많은 AI 팀원을 효과적으로 관리하려면 **팀 리드 역량**이 필요하다.

- 여러 AI 에이전트를 동시에 오케스트레이션
- 각 에이전트의 역할과 책임 명확히 정의
- 에이전트 간 협업과 충돌 관리
- 지속적으로 Harness와 Workflow 개선

## Cocrates Harness와의 관계

**Cocrates = Co(함께) + Socrates(소크라테스)**

Cocrates Harness는 단순한 AI 도구가 아니라, **AI를 길들이고 함께 달리는 프레임워크**다. 핵심은 두 방향의 소크라테스 관계:

- **사용자 → AI (소크라테스):** 사용자가 질문하고, AI의 결과물을 검토하고(U→A→E→A), 판단하여 잘못된 생성을 방지 — AI-native Engineer의 핵심 역량
- **AI(Cocrates) → 사용자 (소크라테스):** 사용자가 막연한 요청을 하거나 AI의 산출물을 모른 채 승인하지 않도록 방지. 대충 넘어가는 것을 제대로 이해하도록 하고, 무지를 알게 하여 스스로 역량을 키우도록 돕는 AI 코치

양방향 모두 **"질문 → 검토 → 판단"** 의 소크라테스 방식을 따른다.

### 구체적 메커니즘

- **소크라테스식 학습(Education → Knowledge Capture → Reflection)**: 검토(U→A→E→A) 역량과 구조적 사고를 훈련
- **구조 기반 생성(ADR → Spec → Generation → Verification)**: AI 팀원에게 일을 위임하고 검토하는 워크플로우를 체계화
- **스킬 생성(Skill Creation)**: 반복적인 업무 패턴을 표준화하여 더 많은 AI 팀원을 관리할 수 있는 기반 마련
