# Presentation Definition — Episode 01

## Topic (S1)
같은 LLM, 다른 결과: 당신의 AI는 몇 명분인가요?

## Purpose
이 에피소드는 **AI-assisted Engineer**와 **AI-native Engineer**를 **명확히 구분**하여 제시한다. 두 방식은 스펙트럼이 아니라, AI를 대하는 **방식의 차이**다.

- **AI-assisted Engineer:** AI가 **나를 도와주는** 구조. AI = 보조 도구.
- **AI-native Engineer:** AI가 **일을 수행하고 내가 검토**하는 구조. AI = 팀원. 지시 → 검토(이해 → 분석 → 평가 → 승인).

그리고 AI-native Engineer라면, 다음 질문을 던진다:
> **"당신의 AI는 몇 명의 팀원입니까? 1명? 4명? 10명? 20명? 당신은 몇 명의 팀원을 리딩해서 업무를 수행할 수 있습니까?"**

결국 **당신의 역량**이 중요하다. Cocrates Harness는 당신이 AI-native Engineer가 되도록 하고, 당신의 팀 리드 역량을 키우는 에이전트 하네스다.

> 참고: AI-native Engineer의 상세 정의는 `presentation/docs/ai-native-engineer.md` 참조.

## Audience
- **주 시청자:** 소프트웨어 개발자(초급~중급), AI 도구(ChatGPT, Claude, Cursor 등)를 업무에 사용 중인 사람
- **함께 볼 시청자:** AI 도구를 활용하는 모든 지식 노동자
- **사전 지식:** LLM 사용 경험 (적어도 한 번 이상 AI에게 코드/문서 생성을 요청해본 경험)
- **시청 동기:** "AI를 쓰는데도 왜 내 결과물은 항상 부족하지?"라는 막연한 불만 또는 호기심

## Logistics
- **형식:** 온라인 교육용 동영상 (녹화/배포)
- **예상 길이:** 약 10~12분
- **톤/스타일:** 부드러운 대화체. 전문성을 유지하되 누구나 이해할 수 있는 비유와 사례 중심. 시청자에게 직접 말을 건네는 1:1 코칭 느낌.
- **슬라이드 구성 (고정 템플릿):**
  1. 타이틀 페이지 (이전 맥락 언급 + 소개)
  2. 학습 목표
  3~N. 에피소드 내용 (블로그 스토리라인에 따라 유연)
  N+1. 학습 목표 확인 (소크라테스식 질문/미션)
  N+2. 다음 에피소드 연결

## Success Criteria
- 시청자가 **AI-assisted Engineer와 AI-native Engineer의 차이**를 명확히 이해한다
  - AI-assisted: AI가 나를 도와줌 (도구)
  - AI-native: AI가 일을 수행하고 내가 검토함 (팀원)
- 시청자가 AI-native Engineer의 개념과 필요성을 이해한다
- 시청자가 "직접 코딩하지 않고도 프로덕션 수준의 서비스를 출시할 수 있다"는 OpenAI 사례를 통해 가능성을 확신한다
- 시청자가 스스로에게 **"당신의 AI는 몇 명의 팀원입니까?"** 라는 질문을 던지고 답한다
- 시청자가 Cocrates Harness가 AI-native Engineer가 되고 팀 리드 역량을 키우는 도구임을 이해한다
- 학습 목표 확인 단계(N+1)에서 시청자가 **"지금 나는 AI-assisted인가, AI-native인가?"** 를 스스로 진단할 수 있다

### 흐름 제안

1. **도입 (슬라이드 1~2):** 타이틀 + 학습 목표
   - 학습 목표 예시:
     - AI-assisted vs AI-native의 차이를 이해한다
     - AI-native Engineer의 개념과 가능성을 이해한다

2. **내용 (슬라이드 3~N):**
   - **(슬라이드 3) 문제 제기:** "같은 LLM, 다른 결과" — 블로그 원문의 Hook
   - **(슬라이드 4~5) A님 사례 & B님 사례:** 블로그 원문의 A/B 대비
   - **(슬라이드 6) AI-assisted vs AI-native 명확한 구분 & "가능할까?":**
     - AI-assisted: AI가 도와주는 구조 (도구)
     - AI-native: AI가 일을 수행하고, 내가 leader로 AI 팀원을 이끄는 구조 (팀원)
     - 검토 = 이해 → 분석 → 평가 → 승인
     - 하지만... "이게 진짜 가능할까?"라는 의문 제기
   - **(슬라이드 7) 실제 사례 — OpenAI (증명):** "가능하다" — 직접 코딩하지 않고 100만 줄 프로덕션 코드 출시
   - **(슬라이드 8) 핵심 질문:** "당신의 AI는 몇 명의 팀원입니까?" — 가능함을 확인했으니, 이제 당신의 역량이 중요하다
   - **(슬라이드 9) Cocrates Harness — Co-Socrates 의미:** Cocrates = Co(상호) + Socrates. 사용자와 AI가 서로 소크라테스가 되는 "상호 소크라테스 관계"가 핵심. 사용자는 AI를 검토·판단하고, AI는 사용자가 막연한 요청/무지한 승인을 하지 않도록 방지하여 스스로 역량을 키우게 함. 이것이 AI-native Engineer로서 AI 팀원을 리딩하는 방법.
   - **(슬라이드 10) 핵심 요약:** (1) 같은 LLM, 다른 결과 — 방식의 차이 (2) AI-native Engineer — 지시→검토→승인 (3) "당신의 AI는 몇 명의 팀원인가?" — 당신의 역량이 중요하다

3. **학습 목표 확인 (슬라이드 N+1):**
   - 질문: "지금 나는 AI-assisted인가, AI-native인가?"
   - 미션: 오늘 자신의 AI 사용 패턴을 1가지 떠올려보기

4. **다음 에피소드 연결 (슬라이드 N+2):**
   - "AI-native Engineer의 핵심 역량은 검토(U→A→E→A)이며, 그 원칙을 다음 편에서 선언한다"
   - 다음 편: "The Unexamined Code Is Not Worth Generating"

## Constraints & Taboos
- **범위 안:** AI-assisted vs AI-native 명확한 구분, A님/B님 사례 대비, OpenAI 실사례, Cocrates Harness 소개
- **범위 밖:** Cocrates Harness의 설치 방법(Ep3), 구체적인 스킬 설명(Ep7~12), 다른 도구와의 성능 비교
- **금지:** "AI를 쓰지 마라"는 부정적 메시지. AI 자체를 폄하하지 않고, 사용자의 태도 변화를 권장하는 방향
- **기술적 상세:** 지나치게 깊은 기술 설명은 피하고, 누구나 공감할 수 있는 사례에 집중
