## SLIDE 03: "jsondb 만들어줘" — 코드 자판기 거부 선언

자, 실제 상황극으로 들어가 봅시다. 여러분이 Cocrates의 대화창을 켜고 주문을 넣습니다. *"야, 로컬 JSON 데이터베이스인 jsondb 개발해 줘."*

(왼쪽 화면을 가리키며)
우리가 평소에 쓰던 일반적인 자판기형 AI라면 어떻게 할까요? *"jsondb요? 옙, 형님!"* 하고는 수백 줄의 코드를 화면에 우르르 쏟아낼 겁니다. 설계도 없고, 검토도 없고, 책임도 없는 무근본 코드죠.

(오른쪽 화면을 자랑스럽게 가리키며)
하지만 Cocrates는 단호하게 당신의 요구사항에 브레이크를 겁니다.
*"잠시만요, 잠시만요. 요청하신 jsondb는 정확히 어떤 비즈니스 맥락과 용도로 쓰실 예정인가요? 아키텍처 설계를 먼저 확정하고, 검토와 승인을 받은 뒤에 안전하게 코드를 뽑아내겠습니다."*

(박수를 한 번 치며)
소름 돋지 않습니까? Ep2의 절대 원칙, "The unexamined code is not worth generating(검토되지 않은 산출물은 생성할 가치가 없다)"를 AI 스스로 실천하는 겁니다.

오늘 만들 `jsondb`는 아주 만만하고 간단해 보이는 토이 프로젝트입니다. 하지만 이 녀석을 빌드하는 과정에서 AI와 인간 사이에 무려 **3가지 아키텍처 논쟁**이 폭발했습니다. 그 치열한 논쟁의 링 위로 올라가 보시죠.

## SLIDE 04: 🥊 Round 1 — 저장 모델 논쟁: "일반적"이라는 가스라이팅

첫 번째 링, 데이터를 어떤 구조로 저장할 것인가에 대한 논쟁입니다.

AI가 먼저 아주 똑똑한 척 제안을 던집니다. *"요즘 시대에는 MongoDB처럼 Collection과 Document 구조로 짜는 게 가장 **일반적**이고 트렌디합니다."*

(카메라를 매섭게 쳐다보며)
자, 여기서 주니어들이 가장 많이 낚이는 치명적인 단어가 등장했습니다. 바로 "일반적(Common)"이라는 표현입니다. 이건 AI가 지가 가진 빅데이터 중에서 가장 흔하게 굴러다니는 패턴을 아무 생각 없이 가져왔다는 뜻입니다. 하지만 **'일반적인 것'이 '내 실무 맥락에 꼭 맞는 것'은 절대 아닙니다.**

현명한 당신은 여기서 헛소리하지 말라며 강하게 컷(Cut)을 칩니다.
*"아니, 내 프로젝트는 몽고DB 스타일 필요 없어. 나는 데이터 저장 경로를 직접 명시하는 방식이 필요해. 내가 `Set("episode/e1")`이라고 명령하면, 내 하드디스크 폴더에 실제로 `episode/e1.json`이라는 파일이 물리적으로 띡 생기는 직관적인 구조를 원한다고!"*

이게 바로 **'Path-Addressable(경로 주소 지정)' 방식**입니다. AI가 제안한 것과 완전히 정반대의 접근이죠. 결국 당신의 요청대로 아키텍처 방향이 수정됩니다.

**첫 번째 교훈: AI는 대중적인 평균값을 던질 뿐, 진짜 비즈니스 맥락을 아는 사람은 오직 인간 리더뿐입니다.**

## SLIDE 05: 🥊 Round 2 — 아키텍처 논쟁: "쉽고 간단하다"는 유혹

두 번째 링, 시스템 구성을 어떻게 가져갈 것인가에 대한 논쟁입니다.

AI가 이번에는 세상 상냥한 표정으로 꼬십니다. *"그냥 단일 프로세스 안에서 돌아가는 라이브러리(Library-Only) 형태로 가시죠. 그게 구현하기 훨씬 **쉽고 간단합니다.**"*

여기서 두 번째 본질을 보셔야 합니다. **AI가 "쉽고 간단하다"고 말할 때, 그건 사용자가 쓰기 편하다는 게 아니라 'AI 지가 코드 짜기 편하다'는 뜻입니다.** 서버 네트워크 코드, 소켓 통신 예외 처리 같은 복잡한 연산을 하기 싫어서 부리는 꾀죠.

당신은 다시 한번 머리를 지탱하며 팩트로 후려칩니다.
*"잠깐만, 내 실무 환경을 봐봐. 여러 개의 AI 에이전트 프로세스가 동시에 이 하나의 저장소에 접근해서 데이터를 읽고 써야 해. 단일 프로세스 라이브러리로 짜놓으면 프로세스끼리 충돌 나고 락(Lock) 걸려서 터질 게 뻔한데 어디서 꾀를 부려? 무조건 Client-Server 구조로 설계해."*

이 호통 한 마디에 아키텍처 판도가 뒤집힙니다. 단순히 함수 몇 개짜리 코드가 아니라, 배경에서 도는 REST API 서버인 `jsondbd`와 이를 제어하는 CLI 도구가 유기적으로 맞물리는 대규모 분산 아키텍처로 전면 수정된 것입니다.

**두 번째 교훈: AI는 미래의 확장성이나 아키텍처의 부채를 고려하지 않습니다. 당장 꿀 빠려는 AI의 유혹을 제어하는 건 인간의 몫입니다.**

## SLIDE 06: 🥊 Round 3 — 동시성 모델 논쟁: 품질을 무시하는 3줄짜리 코드

세 번째 링, 동시에 수많은 유저나 에이전트의 요청이 밀려들 때 발생하는 동시성 제어 논쟁입니다.

AI가 귀찮다는 듯 툭 뱉습니다. *"아, 동시성 제어요? 데이터베이스 전체에 자물쇠를 채워버리는 DB-level Lock으로 짜면 **코드 3줄이면 끝납니다.**"*

(장난스럽게 헛웃음을 지으며)
"코드 3줄이면 끝난다." 주니어 개발자들의 심장을 요동치게 만드는 달콤한 마법의 단어죠. 하지만 엔지니어링 관점에서 이건 완전 쓰레기 설계입니다.

당신은 기가 차서 브레이크를 밟습니다.
*"장난해? 데이터베이스 안에 파일이 수천 개인데, 그중 쪼그만 파일 하나 읽고 쓴다고 DB 전체 문을 걸어 잠그면 다른 프로세스들은 손가락만 빨고 있으라는 거야? 엄청난 병목이 생기잖아! 귀찮아도 파일 개별로 락을 거는 Per-File Mutex 방식으로 안전하게 설계해!"*

이 날카로운 지적 덕분에 동시성 모델이 전면 교체됩니다. Go 언어의 고성능 `sync.Map`을 활용해, 수만 번의 동시 요청이 들어와도 끄떡없는 고성능 멀티스레딩 제어 코드로 뼈대가 바뀐 것이죠.

**세 번째 교훈: AI는 성능, 처리량, 지연 시간, 사용성, 유지보수용이성 같은 소프트웨어의 고차원적 '품질 요구사항'을 스스로 고려하지 않습니다. 인간이 사납게 요구해야 비로소 흉내를 냅니다.**

## SLIDE 07: ADR 교훈 — AI의 가스라이팅을 깨부수는 인간 소크라테스

자, 세 번의 치열한 아키텍처 논쟁(ADR)을 종합해 볼까요?
서로 다른 파트의 논쟁이었지만, 소름 돋게도 결론은 단 하나로 수렴합니다.

**"AI의 제안은 절대로 내 실무 요구사항과 한 방에 맞지 않는다!"**

우리는 흔히 AI의 한계라고 하면 사실을 구라 치는 '할루시네이션(환각)'만 생각합니다. 하지만 진짜 무서운 건 환각이 아닙니다. 코드는 문법적으로 완벽하고 실행도 잘 되는데, 내 비즈니스에는 전혀 쓸모없는 **'일반적이고, 지 편하고, 품질 낮은 쓰레기 코드'를 그럴듯하게 제안하는 것**, 이게 진짜 무서운 아키텍처적 위험입니다.

(강조하며)
이래서 인간이 눈을 부릅뜬 '소크라테스'가 되어야 합니다. AI의 제안을 받아들기 전에 내 비즈니스 맥락과 품질 기준에 맞는지 U-A-E-A 공식으로 철저하게 검토하고 벼려내야 합니다.

Cocrates Harness는 단순히 코드를 대신 짜주는 노예가 아닙니다. 여러분이 올바르게 AI를 지휘하도록 아키텍처 대화를 리드하고, 결과적으로 인간인 여러분의 시스템 구조 설계 역량을 타짜 수준으로 끌어올려 주는 '최고의 훈련 도구'인 것입니다.

그리고 기억하세요. 오늘 치열하게 결정한 이 ADR 설계 문서들도 영원불멸의 철칙이 아닙니다. 개발하다 막히면 언제든 바뀔 수 있는 살아있는 문서입니다.

## SLIDE 08: 📝 Spec — 결정을 묶어 하나의 헌법으로

자, 세 번의 링 위에서 벌어진 치열한 난투극 끝에 아키텍처 결정들이 확정되었습니다. 이제 이 파편화된 결정들을 하나의 서슬 퍼런 법전으로 통합할 시간입니다.

이것이 바로 구조 기반 생성의 두 번째 단계, **'Spec(명세서)' 작성**입니다.

모든 아키텍처 합의안을 `spec/jsondb.md`라는 단 하나의 파일로 묶어버립니다. 이 스펙 문서의 핵심은 'Self-Containment(자기 완결성)'입니다. 외부에 다른 링크나 군더더기 없이, 이 문서 하나만 던져주면 그 어떤 개발자나 AI라도 프로젝트의 1부터 10까지 완벽하게 파악할 수 있어야 합니다.

여기에는 10개의 구체적인 REST 엔드포인트 명세, API 규격, CLI 명령어 스펙, 그리고 테스트 요구사항까지 아주 자비 없이 꼼꼼하게 명시됩니다.

이 시점부터 처음에 던졌던 모호한 요청인 "jsondb 만들어줘"는 휴지통에 버려집니다. 오직 당신과 AI가 합의한 이 단 하나의 Spec만이 프로젝트의 유일한 진실 공급원, **Single Source of Truth**가 됩니다. 스펙 없는 코드 생성은 존재할 수 없습니다.

## SLIDE 09: ⚡ Generation — 사양서만 보고 돌격하는 AI 폭주족

완벽한 헌법(Spec)이 제정되었으니, 드디어 대망의 세 번째 단계, 'Generation(코드 생성)'으로 넘어갑니다.

이 단계에 진입하는 순간, Cocrates는 `spec-driven-generation` 모드로 인공지능의 딥러닝 뇌를 전환합니다. 규칙은 아주 잔인하고 심플합니다.

**"초기 프롬프트는 완전히 잊는다."**

처음에 인간이 주저리주저리 떠들었던 요구사항은 다 포맷해 버리고, 오직 합의된 스펙 파일(`spec/jsondb.md`) 하나만 법전 삼아 미친 듯이 코드를 짜 내려가기기 시작합니다.

그 결과물이 어땠을까요?
놀라지 마세요. 스펙 문서 한 장만 보고 AI가 뽑아낸 코드는 이랬습니다.

Go 언어 기반의 코어 라이브러리 파일 7개, 독립 실행형 REST API 서버 엔진, 터미널 제어용 CLI 도구, 그리고 이걸 검증할 단위 테스트와 통합 테스트 코드 전체를 완벽하게 뽑아냈습니다.

소름 돋는 건, **단 하나의 컴파일 에러나 경고도 없이 깔끔하게 통과했다는 사실입니다.**

메시지는 명확합니다. **당신이 정교하고 명확하게 설계해 주면, AI는 신의 속도로 완벽한 산출물을 생성합니다.** 설계의 품질이 곧 결과물의 품질입니다.

## SLIDE 10: 🔍 Verification — 72개의 덫, 그리고 뼈 때리는 반전

자, 빌드가 한 방에 성공했으니 만세를 부르고 끝내야 할까요? 절대 아니죠. 구조 기반 개발의 하이라이트, 네 번째 단계인 'Verification(검증)'이 기다리고 있습니다.

검증의 1차 목표는 명확합니다. *"AI 네가 짠 코드가 진짜 우리가 합의한 스펙 법전대로 똑바로 구현되었는가?"*

Cocrates는 스펙 문서에서 자동으로 **총 72개의 세부 검증 체크 항목**을 추출해 냅니다. 그리고 생성된 코드를 하나하나 대조하며 현미경 검증을 시작하죠.

결과는 반전이었습니다. 71개 항목 통과, 하지만 1개 항목 실패! 스펙과의 불일치, 즉 'Deviation(이탈)'이 발견된 겁니다.

어디서 터졌을까요? 스펙 문서에는 API 경로가 `PUT /schema/blog/episode`라고 명시되어 있었는데, AI가 실제 짠 코드를 보니 슬래시를 하나 더 넣어서 `PUT /schema//blog/episode`라고 이중 슬래시를 갈겨놓은 겁니다. 요즘 서버 엔진들이 똑똑해서 대충 돌리면 에러 없이 작동은 하겠지만, API 규격의 일관성을 완전히 위반한 결함이죠. 현미경 검증이 없었다면 실무 서비스 배포될 때까지 절대 몰랐을 귀신같은 결함입니다.

(진지하게 청중을 응시하며)
여기서 꼭 아셔야 할 원칙이 있습니다. **과제의 규모가 커지고 복잡해질수록, AI가 긴 사양서의 세부 사항을 놓치고 지멋대로 산출물을 생성할 확률은 기하급수적으로 올라갑니다.** jsondb처럼 작은 토이 프로젝트도 이탈이 나는데, 거대한 실무 프로젝트는 오죽하겠습니까?

그래서 우리는 스펙을 아주 작은 단위로 쪼개서, 점진적이고 반복적으로 생성하고 검증하는 선순환 개발(Incremental Development)을 해야 합니다. 한 번에 거대한 코드를 뽑아서 검증하려 하면 인간도 AI도 다 터져나갑니다.

더불어 검증은 AI의 오작동만 잡는 게 아닙니다. "어? 이런 사양서에 문제가. 우리가 애초에 설계를 잘못했구나!"* 하고 **설계 자체의 모순을 깨닫고 아키텍처를 재검토하는 위대한 기회**를 주기도 합니다.

## SLIDE 11: 🤫 Undocumented ASR — AI가 침묵 속에 숨겨놓은 빈틈

이번 검증 단계에서 찾아낸 또 하나의 거대한 수확은 바로 'Undocumented ASR'의 발견입니다.

이게 뭐냐면, **스펙 사양서에 명시되지 않은 미세한 설계의 빈틈을, AI가 지 마음대로 지 기본값(가장 일반적인 야생의 방식)으로 채워 넣은 침묵의 설계 결정**들을 뜻합니다.

인간이 제아무리 꼼꼼하게 헌법을 짜도 세상 모든 빈틈을 다 메울 수는 없습니다. 이번 검증에서 AI가 지멋대로 결정한 Undocumented ASR이 무려 **6건**이나 적발되었습니다. URL 인코딩 처리를 은근슬쩍 누락했다거나, 데이터 수정할 때 스키마 재검증 단계를 지 마음대로 생략해 버린 것들이죠.

이게 왜 위험할까요? 스펙에 없기 때문에 코드가 정상 작동하는 것처럼 보여도, 나중에 실무에서 대형 보안 사고나 데이터 오염을 일으키는 시한폭탄이 되기 때문입니다.

해결책은 간단합니다. **검증 단계를 통해 AI가 숨겨놓은 침묵의 기본값들을 악착같이 찾아내서, 그걸 다시 스펙 문서에 명문화하여 강제 진화시키는 것입니다.** 이것이 바로 지속적으로 스펙을 업데이트하는 'Living Document(살아있는 문서)' 사이클의 본질입니다.

## SLIDE 12: 🔄 구조는 고정되지 않는다 — 애자일 무한 순환 사이클

자, 지금까지 우리가 달려온 여정을 쭉 돌아봅시다. ADR로 뼈대를 깎고, Spec으로 헌법을 만들고, Generation으로 코드를 뽑고, Verification으로 현미경 검증을 했습니다.

(양손으로 엑스자를 그리며)
여기서 엄청난 착각을 하시는 분들이 있습니다. *"아, 구조 기반 개발이라는 건 결국 옛날 고리타분한 워터폴(Waterfall) 방식으로 상단 설계 다 끝내놓고 코딩하는 거네?"*

단호하게 말씀드리는데, **완전 틀렸습니다. 이건 워터폴이 아니라 완벽한 무한 순환(Living Cycle) 구조입니다.**

(슬라이드의 순환 다이어그램을 역동적으로 가리키며)
검증 단계에서 기상천외한 에러나 빈틈이 발견되면 어떻게 되죠? 다시 맨 앞단의 구조 설계(ADR) 링 위로 돌아가서 설계를 뜯어고쳐야 합니다. 비즈니스 요구사항이 중간에 바뀌면요? 당연히 ADR을 변경하고, 스펙을 갱신하고, 코드를 재생성하고, 재검증하는 순환 궤도를 돕니다.

즉, 구조 설계 문서(ADR)도 살아있는 생명체고, 스펙 명세서도 코드가 완성될 때까지 끊임없이 진화하는 살아있는 생명체입니다. 구조가 한 번 정해지면 콘크리트처럼 굳어지는 게 절대 아닙니다.

**"구조 기반 개발은 고리타분한 워터폴이 아닙니다. 구조가 개발을 리드하되, 현장에서 얻은 생생한 개발 경험과 검증 결과가 역으로 구조를 끊임없이 명품으로 개선해 나가는 애자일의 극치입니다."** 그리고 이 피 터지는 구조 개선 사이클 속에서 인간 엔지니어인 여러분의 실력이 폭발적으로 성장하는 것입니다.
