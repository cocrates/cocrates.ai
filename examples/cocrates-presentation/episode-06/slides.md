## 본론 (Slides 3-12) — 약 15분

### Slide 3: "보고서 써줘" 대신 "보고서 쓰는 법"을 AI에게 가르친다
- **Section:** 본론 — 시작 (2.1)
- **Purpose/goal:** 일회성 요청과 스킬의 차이를 극명하게 보여준다. "물고기를 잡아달라 vs 물고기 잡는 법을 가르쳐달라" 비유로 스킬의 가치를 전달한다.
- **Content:**
  - **좌측: 일반적인 접근 (일회성 요청)** — "보고서 써줘"
    - 매번 새로운 요청, 매번 다른 품질, 반복 피로
    - AI가 그때그때 다르게 응답 → 일관성 없음
  - **우측: 🦉 Cocrates의 접근 (스킬)** — "보고서 자동화 스킬 만들어줘"
    - **한 번 만들면 영원히 재사용**
    - 같은 유형의 요청 → 항상 같은 워크플로우로 진행 → 일관된 품질
  - **비유:** "물고기를 잡아달라" vs "물고기 잡는 법을 가르쳐달라"
  - **핵심:** 스킬 = **AI의 행동 규범(Standard Operating Procedure)**. 같은 유형의 요청에 대해 AI가 일관된 방식으로 작업하도록 만든다
  - **오늘 만들 것:** `document-authoring` 스킬 — 보고서/설명문 자동화 워크플로우
- **Type:** 사례/비교

### Slide 4: 🎯 Kernel — 핵심 목적을 단 한 문장으로 압축
- **Section:** 본론 — Kernel 단계 (2.2)
- **Purpose/goal:** 스킬 설계의 첫 단계인 Kernel의 중요성을 전달한다. 모호한 요청을 단 한 문장으로 압축하는 것이 스킬 설계의 첫걸음임을 이해시킨다.
- **Content:**
  - **Cocrates의 질문 3종 세트 (모호함을 구체화):**
    1. "어떤 종류의 보고서를 주로 쓰실 건가요?"
    2. "주요 독자는 누구인가요?"
    3. "산출물은 어떤 파일 형태로 저장되나요?"
  - **사용자의 응답:** "일반적인 문서, 누구나 독자, .md 파일"
  - **Kernel 수립 (스킬의 헌법):**
    > "이 스킬은 Markdown 형식의 설명문/보고서를 검토 가능한 단계를 거쳐 생성하도록 돕는다."
  - **💡 포인트:** Kernel은 스킬의 **헌법**이다. 이 한 문장이 모든 결정의 기준이 된다. 모호한 요청을 단 한 문장으로 압축하는 것, 이것이 스킬 설계의 첫걸음이다.
- **Type:** 개념/사례

### Slide 5: 🧱 Frame — 산출물의 뼈대와 작업 파일 트리
- **Section:** 본론 — Frame 단계 (2.3)
- **Purpose/goal:** Snowflake 점진적 확장을 위한 파일 트리와 중간 산출물 관리 방식을 전달한다. "한 번에 통으로 쓰지 않는다"는 원칙을 이해시킨다.
- **Content:**
  - **문서 레이아웃:** 메타정보 → 개요 → 본문(섹션/서브섹션) → 결론 → 부록
  - **Snowflake 파일 트리 (중간 산출물 격리 저장):**
    ```
    docs/{slug}/
    ├── outline.md              # 1단계: 문서 개요 및 메타정보
    ├── sections.md             # 2단계: 구조 도면
    ├── sections/               # 3단계: 섹션별 본문 저장소
    └── {slug}.md               # 4단계: 최종 통합본
    ```
  - **Cocrates의 제안:** "한 번에 통으로 쓰는 것보다 섹션 단위로 쪼개어 검토하며 작성하는 게 안전하겠죠?"
  - **💡 포인트:** Snowflake의 핵심 — **작은 단위로 쪼개고, 단계별로 파일에 저장하고, 검토하고, 승인받고, 다음 단계로**. 채팅창에서 모든 걸 주고받지 않는다. 파일이 작업의 단위다.
- **Type:** 개념/구조

### Slide 6: 🤯 Outline — AI가 가장 쉽고 일반적인 결정을 내리다
- **Section:** 본론 — Outline 단계 (2.4)
- **Purpose/goal:** AI가 "당연히"라고 생각하는 것이 사실은 **가장 쉽고 일반적인 선택**에 불과함을 보여준다. Ep5의 ADR 교훈(일반적이다/쉽고 간단하다/품질 미반영)이 스킬 생성에서도 동일하게 적용됨을 전달한다.
- **Content:**
  - **상황:** 절차적 동선(P1~P5 단계) 구체화 중
  - **AI(Cocrates)의 무심한 제안:**
    > "문서를 작성할 때는 당연히 **서론 → 본론 → 결론** 순서로 순차적으로 뽑아내는 게 자연스럽겠죠?"
  - **AI의 가정:** 모든 문서는 서론→본론→결론 구조여야 한다
  - **왜 이런 제안을 했을까?** AI가 가장 많이 학습한 패턴이고, 구현하기 가장 쉬운 구조이기 때문
  - **Ep5 연결:** 이것은 **AI가 일반적이고 쉬운 결정을 내린 것**이다. Ep5의 ADR 교훈과 같은 맥락 — AI의 제안은 항상 내 요구와 맞는 것은 아니다.
- **Type:** 사례/극적 전환

### Slide 7: 💥 AI의 제안을 검토 없이 수용하지 마라 — U→A→J→A
- **Section:** 본론 — Outline 단계 (2.4)
- **Purpose/goal:** 가장 중요한 교훈을 전달한다. AI가 제안한 것이 "당연해 보인다"고 해서 검토 없이 받아들이면 안 된다. 사용자의 맥락에 맞는 더 나은 선택을 위해 반드시 **이해→분석→판단→승인** 단계를 거쳐야 한다.
- **Content:**
  - **사용자의 브레이크:**
    > "**항상 그런 순서로 정형화된 건 아니야.** 글의 목적에 따라 스토리 중심의 기승전결이 될 수도 있고, 문제 해결 중심의 구성이 될 수도 있어."
  - **이 결정의 의미:**
    - AI가 제안한 것은 가장 **일반적이고 쉬운** 결정 (서론→본론→결론)
    - 사용자가 내린 것은 **자신의 맥락에 맞는** 결정, 즉 **더 복잡한 요구사항**
    - 단순 수용이 아니라 **이해→분석→판단→승인(U→A→J→A)** 의 결과
  - **💡 핵심 교훈: AI의 제안을 검토 없이 수용하는 것은 좋지 않다.**
    - AI가 "당연히"라고 생각하는 것이 항상 옳은 것은 아니다
    - **반드시 이해→분석→판단→승인(U→A→J→A)** 의 단계를 거쳐야 한다
    - 이것이 Cocrates의 설계 심사 원칙이다 — AI가 일을 수행하고, 사람이 리더로서 결정한다
  - **Ep5와의 연결:** Ep5의 ADR 3가지 교훈(일반적이다/쉽고 간단하다/품질 미반영)과 같은 패턴. AI의 제안이 가장 쉽고 일반적인 선택일 뿐임을 인식하고, 사용자의 맥락에서 더 나은 방향으로 이끈다.
- **Type:** 사례/극적 전환

### Slide 8: 📜 Spec — 룰북과 금지 조항
- **Section:** 본론 — Spec 단계 (2.5)
- **Purpose/goal:** 사용자의 통찰이 반영된 Spec이 어떻게 구체적인 행동 규범으로 정리되는지 보여준다. Do(해야 할 일)와 Don't(하지 말아야 할 일)를 함께 명시하는 Spec의 완성도를 전달한다.
- **Content:**
  - **6가지 맞춤형 구성 방식 선택지 (사용자의 통찰이 반영됨):**
    - 설득/제안: 서론-본론-결론
    - 스토리/사례: 기승전결
    - 문제 해결: 문제-원인-해결-제안
    - 비교/분석: 기준-대안-평가-결론
    - (기타 2가지)
  - **7가지 금지 사항 (Anti-Patterns):**
    - 승인 게이트 통과 전 본문 작성 금지
    - 중간 산출물을 파일로 저장하지 않고 채팅창에만 남기지 말 것
    - 등
  - **💡 포인트:** Spec은 **합의된 행동 규범**이다. 무엇을 해야 하는지(Do)뿐 아니라, 무엇을 하지 말아야 하는지(Don't)도 명시한다.
- **Type:** 개념/명세

### Slide 9: 🛠️ Skill 생성 — Spec이 코드가 되다
- **Section:** 본론 — Skill 생성 (2.6)
- **Purpose/goal:** Spec이 실제 스킬 파일로 생성되는 순간을 보여준다. 이것이 단순한 문서 작성이 아니라 '지속 가능한 자산'을 만드는 과정임을 전달한다.
- **Content:**
  - **생성:** Spec → `.opencode/skills/document-authoring/SKILL.md` 정식 생성
  - **이것이 의미하는 것:**
    - 더 이상 "보고서 써줘"라고 매번 요청할 필요 없음
    - 같은 유형의 요청이 오면 AI가 **자동으로 이 스킬의 워크플로우를 따라감**
    - **지속 가능한 자산:** 한 번 만들면 영원히 재사용
  - **💡 포인트:** 스킬은 코드와 같다. 한 번 작성하면 계속 사용할 수 있는 **소프트웨어 자산**이다.
- **Type:** 개념/결과

### Slide 10: ✅ Verification — 7개 항목 검증
- **Section:** 본론 — Verification (2.6)
- **Purpose/goal:** 생성된 스킬도 검증되어야 신뢰할 수 있음을 전달한다. Ep5의 Verification이 스킬 생성에서도 동일하게 적용됨을 보여준다.
- **Content:**
  - **원칙:** 생성된 스킬이 자체 검증(Verification)을 통과해야 함
  - **7개 검증 항목 (전원 PASS ✅✅✅✅✅✅✅):**
    1. 최종본보다 중간 산출물을 먼저 강제하는가? ✅
    2. 사용자 승인 게이트가 확실히 작동하는가? ✅
    3. 완료 조건과 금지 사항이 구체적인가? ✅
    4~7: (추가 항목) ✅
  - **💡 포인트:** Ep5에서 배운 Verification이 여기서도 적용된다. 스킬 자체도 검증되어야 신뢰할 수 있다. 검증은 코드에만 있는 것이 아니다.
- **Type:** 사례/분석

### Slide 11: 🔄 "보고서 써줘" 한 마디의 의미가 바뀌었다
- **Section:** 본론 — 스킬 완성 후 (2.7)
- **Purpose/goal:** 스킬이 만들어지기 전과 후의 차이를 극명하게 비교한다. 사용자의 역할이 '수동적 수용자'에서 '능동적 설계자'로 바뀌었음을 강조한다.
- **Content:**
  - **스킬 전:** "블록체인 시장 조사 보고서 써줘"
    - AI가 웹 검색 내용을 짜깁기한 글을 던짐
    - 품질을 예측할 수 없음, 매번 새로운 검증 필요
  - **스킬 후:** "블록체인 시장 조사 보고서 써줘"
    - 스킬의 룰에 따라 **단계별로 진행**
    - **[P1]** "타겟 독자와 목적부터 정하시죠. outline.md를 승인해 주세요."
    - **[P2]** "기술 분석 위주니 '비교/분석' 포맷이 좋겠습니다. 동의하십니까?"
    - **[P3]** "1번 섹션을 작성했습니다. 검토 의견을 주세요."
  - **핵심:** 스킬이 만들어지면 **사용자의 역할이 '수동적 수용자'에서 '능동적 설계자'로 바뀐다.**
- **Type:** 사례/비교

### Slide 12: ❄️ Snowflake 점진적 확장의 힘
- **Section:** 본론 — Snowflake (2.7)
- **Purpose/goal:** Ep4, Ep5, Ep6에 공통으로 적용된 Snowflake 방법론을 정리한다. 작게 시작해서 점진적으로 확장하는 공통 패턴임을 깨닫게 한다.
- **Content:**
  - **Snowflake 원칙:**
    1. 작게 시작한다 (Kernel 한 문장)
    2. 단계별로 확장한다 (Frame → Outline → Spec)
    3. 각 단계에서 검토와 승인을 받는다
    4. 중간 산출물을 파일로 저장한다 (채팅창 유실 방지)
  - **Ep4-5-6의 공통 방법론:**
    - Ep4 (학습): Education → Knowledge Capture → Reflection
    - Ep5 (생성): ADR → Spec → Generation → Verification
    - Ep6 (스킬): Kernel → Frame → Outline → Spec → Skill → Verification
    - **모두 Snowflake: 작게 시작해서 점진적으로 확장**
  - **💡 포인트:** Snowflake는 Cocrates의 공통 패턴이다. 배움, 생성, 스킬 — 모든 활동에 동일한 방법론이 적용된다.
- **Type:** 개념/종합

## 결론 (Slides 13-14) — 약 3분

### Slide 13: 핵심 요약 + 학습 확인
- **Section:** 결론
- **Purpose/goal:** 발표의 핵심을 4가지 포인트로 요약하고, 시청자가 스스로 이해를 확인할 수 있도록 자기 질문을 제시한다.
- **Content:**
  - **4-Point 핵심 요약:**
     1. **🔄 스킬 = AI의 일하는 방식을 설계하는 것.** 일회성 요청을 넘어, AI의 행동 규범(SOP)을 정의한다. 한 번 만들면 영원히 재사용 가능한 지속 가능한 자산이다.
     2. **🔍 AI의 제안을 검토 없이 수용하지 마라.** AI는 가장 쉽고 일반적인 결정을 내린다. 사용자의 맥락에 맞는 더 나은 선택을 위해 반드시 **이해→분석→판단→승인(U→A→J→A)** 단계를 거쳐야 한다. 이것이 Cocrates의 설계 심사 원칙이다.
     3. **❄️ Snowflake는 공통 패턴이다.** 작게 시작해서 점진적으로 확장하고, 단계별로 검토하고 승인받는 방식은 학습, 생성, 스킬 모든 활동에 적용된다.
     4. **👑 스킬이 만들어지면 사용자는 능동적 설계자가 된다.** 더 이상 AI가 주는 결과물을 수동적으로 기다리지 않고, 스스로 설계하고 통제한다.
