# Script: Slide 06 — ADR ③ — 동시성 모델 논쟁

**Time:** ~1.5분

**[SLIDE 06: ADR ③ 동시성 모델 논쟁]**

**상황:** "동시 접근을 어떻게 제어할까?"
**AI의 제안:** "DB 전체 락(DB-level Lock)이면 코드 3줄이면 끝납니다"
→ **핵심 문제: 품질을 충분히 반영하지 못한다 (Quality-blind)**
**사용자의 브레이크:** "파일 하나 읽을 때 전체 DB를 멈추면 병목이잖아! 파일별 락(Per-File Mutex)으로 해줘."
**결과:** `sync.Map`을 활용한 고성능 동시성 제어
**💡 교훈:** AI는 품질을 충분히 반영하지 못한다.

---

세 번째 논쟁입니다. 동시에 여러 요청이 들어올 때 어떻게 제어할까?

AI의 제안을 들어보세요. "DB 전체 락, DB-level Lock이면 코드 3줄이면 끝납니다."

**"코드 3줄이면 끝납니다."** 이 말 속에 AI의 특징이 잘 드러나 있습니다. 구현이 얼마나 쉬운지를 강조하고 있죠. 하지만 과연 그게 좋은 설계일까요?

사용자가 브레이크를 밟습니다.

"생각해보세요. 파일 하나를 읽기 위해 전체 DB를 멈추면 어떻게 될까요? 엄청난 병목이 생길 겁니다. 파일별로 락을 거는 Per-File Mutex 방식으로 해주세요."

이 지적에 따라 동시성 모델이 완전히 바뀝니다. Go 언어의 `sync.Map`을 활용한 고성능 동시성 제어로 전환된 거죠.

(잠시 멈춤)

여기서 얻는 교훈은 이것입니다. **AI는 품질을 충분히 반영하지 못합니다.** 성능, 처리량, 응답 시간 — 이런 품질 속성은 AI가 스스로 고려하지 않습니다. 사용자가 직접 지적해야 비로소 반영됩니다.

세 번의 논쟁을 통해 우리는 한 가지 중요한 사실을 깨달았습니다. AI의 제안이 항상 내 요구와 맞는 것은 아니라는 점을요. 다음 슬라이드에서 이 내용을 종합해보겠습니다.
