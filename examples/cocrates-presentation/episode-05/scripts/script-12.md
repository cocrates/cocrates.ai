# Script: Slide 12 — 🔄 구조는 고정되지 않는다 — Living Cycle

**Time:** ~1.5분

**[SLIDE 12: 구조는 고정되지 않는다 — Living Cycle]**

**오해: Waterfall**
구조 설계 → 구현 → 검증 (끝)

**현실: Living Cycle**
구조가 개발을 이끌지만, 구조 자체도 계속 진화한다

```
       (새 요구사항 / 검증 발견)
                ↑
구조 설계(ADR) → Spec → Generation → Verification
    ↑                                    ↓
    └────────── (피드백 반영) ←───────────┘
```

**핵심:**
- ADR도 살아있는 문서다
- Spec도 살아있는 문서다
- 구조가 정해지면 변하지 않는 것이 아니다

---

자, 지금까지의 과정을 다시 한번 돌아볼까요?

ADR로 아키텍처를 설계하고, Spec으로 결정을 통합하고, Generation으로 코드를 만들고, Verification으로 검증했습니다.

그런데 여기서 많은 분들이 오해하실 수 있는 점이 있습니다. 바로 **이 과정을 Waterfall로 보는 것**입니다.

Waterfall은 이렇게 생각합니다. "구조 설계하고, 구현하고, 검증하면 끝."

(잠시 멈춤)

하지만 현실은 다릅니다. **이것은 순환(Cycle)입니다.**

검증에서 문제가 발견되면 어떻게 될까요? 구조 설계, 즉 ADR로 돌아가야 합니다. 결정을 재검토하고, 필요하면 ADR 자체를 변경해야 합니다.

요구사항이 변경되면요? 마찬가지입니다. ADR 변경 → Spec 갱신 → 재생성 → 재검증의 순환을 거칩니다.

(슬라이드의 다이어그램을 가리키며)

이 그림을 보세요. 구조 설계(ADR)에서 시작해서 Spec, Generation, Verification으로 이어집니다. 그런데 검증에서 발견된 문제나 새로운 요구사항이 있으면, 다시 구조 설계로 돌아갑니다. ADR 결정이 변경될 수 있고, Spec도 함께 갱신됩니다.

여기서 핵심은 이것입니다.

**ADR도 살아있는 문서입니다.** 오늘 내린 결정이 내일도 유효하라는 법은 없습니다.

**Spec도 살아있는 문서입니다.** ADR이 바뀌면 Spec도 바뀌어야 합니다.

**구조가 한 번 정해지면 변하지 않는 것이 아닙니다.** 구조가 개발을 이끌지만, 개발 경험과 검증 결과가 다시 구조를 개선합니다.

> "구조 기반 개발은 Waterfall이 아니다. **구조가 개발을 이끌고, 개발 경험이 구조를 개선한다.** "
