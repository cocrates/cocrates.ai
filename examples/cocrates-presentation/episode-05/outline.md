#### 슬라이드 2: 학습 목표

- **핵심:** AI의 제안이 항상 옳지는 않다. **사용자가 주도적으로 구조를 설계해야 한다.**
- **학습 목표 (4가지):**

  4. Undocumented ASR(암묵적 설계 결정)의 개념을 이해하고 검증의 중요성을 설명할 수 있다

### 2. 본론 (슬라이드 3~12) — 약 16분

#### 2.1 시작 — 일반 AI와 Cocrates의 차이 (슬라이드 3) — 1.5분

##### 슬라이드 3: "jsondb를 만들어줘" — 하지만 AI는 설계부터 권했다
- **좌측: 일반 AI** — "jsondb? 알겠습니다!" → 수백 줄 코드 와르르
- **우측: 🦉 Cocrates** — "이 jsondb는 어떤 용도인가요? 아키텍처를 먼저 설계하고 검토받은 후 생성하겠습니다."
- **핵심:** Cocrates는 **설계 없는 생성을 거부한다.**
- **오늘 만들 것:** jsondb — 로컬 JSON 저장소. 간단해 보이지만 설계 결정이 많이 필요한 프로젝트.

#### 2.2 ADR 단계 — 3번의 아키텍처 논쟁 (슬라이드 4~7) — 6분

##### 슬라이드 4: ADR ① — 저장 모델 논쟁
- **AI의 제안:** "MongoDB처럼 Collection-Document 구조가 **일반적**이고 쉽습니다"
  - **핵심 문제: 일반적이다 (Generic)**
  - AI가 가장 많이 학습한 패턴 — 수많은 예제에서 봐온 구조
  - 하지만 "일반적"이라는 것은 "보통은 그렇다"는 뜻이지, "내 상황에 꼭 맞다"는 뜻이 아니다
- **사용자의 브레이크:** "내가 지정한 경로가 그대로 파일명이 되는 Path-Addressable 방식이 필요해"
  - 예: `Set("episode/e1.json")` → 실제 `episode/e1.json` 파일
- **결과:** 사용자의 요구대로 경로 매핑 구조로 결정
- **💡 교훈:** **일반적이다 ≠ 내 요구에 맞다.** AI가 가장 흔히 접한 패턴이라고 해서 내 요구사항에 최적인 것은 아니다. 사용자의 구체적인 사용 맥락을 AI는 모른다.

##### 슬라이드 5: ADR ② — 아키텍처 논쟁
- **AI의 제안:** "단일 프로세스 Library-Only가 **쉽고 간단**합니다"
  - **핵심 문제: 쉽고 간단하다 (Simple)**
  - AI 입장에서 구현하기 가장 쉬운 방식 — 서버/네트워크 코드가 필요 없음
  - 하지만 "만들기 쉽다"는 "사용하기 좋다"는 뜻이 아니다
- **사용자의 브레이크:** "여러 AI 에이전트가 동시에 접근할 거야. Client-Server 구조로 가야 해."
- **결과:** REST API 서버(jsondbd) + CLI 도구 포함 아키텍처로 전면 수정
- **💡 교훈:** **쉽고 간단하다 ≠ 올바른 설계다.** AI가 만들기 쉬운 방식이 실제 사용 환경의 요구를 만족하는 것은 아니다. 사용자는 미래의 확장성까지 고려한다.

##### 슬라이드 6: ADR ③ — 동시성 모델 논쟁
- **AI의 제안:** "DB 전체 락(DB-level Lock)이면 코드 3줄이면 끝납니다"
  - **핵심 문제: 품질을 충분히 반영하지 못한다 (Quality-blind)**
  - AI가 코드 3줄로 구현을 끝낼 수 있다고 강조할 정도로 구현은 쉽다
  - 하지만 **성능(처리량, 응답 시간)**이라는 핵심 품질 속성을 전혀 고려하지 않음
- **사용자의 브레이크:** "파일 하나 읽을 때 전체 DB를 멈추면 병목이잖아! 파일별 락(Per-File Mutex)으로 해줘."
- **결과:** `sync.Map`을 활용한 고성능 동시성 제어
- **💡 교훈:** **AI는 품질을 충분히 반영하지 못한다.** 성능, 확장성, 유지보수성 같은 품질 속성은 AI가 스스로 고려하지 않는다. 사용자가 지적해야 반영된다.

##### 슬라이드 7: ADR 교훈 — "AI의 제안은 항상 내 요구와 맞는 것은 아니다"
- **3번의 논쟁 — 각각 다른 문제, 같은 결론:**
  - ADR ① 저장 모델: AI의 제안은 **일반적**이었지만, 내 요구사항과 맞지 않았다
  - ADR ② 아키텍처: AI의 제안은 **쉽고 간단**했지만, 미래 확장성을 담지 못했다
  - ADR ③ 동시성: AI의 제안은 **품질(성능)을 반영하지 못했다**
- **AI 제안의 부적절함은 다양한 형태로 나타난다:**
  - 할루시네이션(Hallucination)만 문제가 아니다
  - **일반적인 패턴**이 내 상황에 안 맞을 수 있다
  - **쉽고 간단한 해결책**이 진짜 필요한 해결책이 아닐 수 있다
  - **품질 요구사항**을 AI가 스스로 고려하지 않을 수 있다
  - → 그러므로 **사용자의 검토와 승인이 필수적이다**
- **Cocrates의 역할:**
  - 단순히 코드를 생성해주는 도구가 아니다
  - **사용자가 올바르게 AI를 사용할 수 있도록** 아키텍처 설계를 대화로 이끈다
  - **사용자의 구조 설계 역량을 키우는 데 도움을 준다**
  - AI의 제안을 무조건 수용하지 않고, 자신의 맥락과 요구사항, 품질 기준에 맞게 **U→A→E→A 검토**하고 방향을 바로잡는 힘을 기른다
- **ADR도 살아있는 문서다:**
  - 오늘의 ADR 결정이 영원한 정답은 아니다
  - 요구사항이 바뀌거나 검증에서 새로운 문제가 발견되면 **ADR 결정도 변경될 수 있다**
  - ADR과 Spec은 함께 진화한다

#### 2.3 Spec 단계 (슬라이드 8) — 1.5분

##### 슬라이드 8: 📝 Spec — 결정을 하나의 헌법으로
- 모든 ADR 결정을 **하나의 설계 명세서**로 통합
- **Self-Containment:** `spec/jsondb.md` 하나만 읽으면 전체 구조 파악 가능
  - 10개 REST 엔드포인트 기능
  - API 시그니처와 CLI 명령어 스펙
  - 테스트 요구사항까지 모두 포함
- **Spec의 역할:**
  - 초기의 모호한 요청("jsondb 만들어줘")은 잊힌다
  - 오직 **통합된 Spec**만이 유일한 진실 공급원(Single Source of Truth)
- **💡 포인트:** Spec 없이 생성은 있을 수 없다. Spec이 곧 **설계의 완성**이다.

#### 2.4 Generation 단계 (슬라이드 9) — 1.5분

##### 슬라이드 9: ⚡ Generation — 오직 Spec만 보고 달린다
- `spec-driven-generation` 모드 활성화
- **원칙:** 초기 프롬프트("jsondb 만들어줘")는 잊고, **합의된 Spec만** 법전으로 삼는다.
- **결과물:**
  - 코어 라이브러리 Go 파일 7개
  - REST API 서버
  - CLI 도구
  - 단위 테스트 + 통합 테스트
  - **컴파일 오류 0개** (`go build`, `go vet` 패스)
- **핵심:** "Spec을 잘 설계하면 AI가 코드를 잘 만든다. Spec의 품질이 결과물의 품질을 결정한다."

#### 2.5 Verification 단계 (슬라이드 10~11) — 3분

##### 슬라이드 10: 🔍 Verification — 설계 검증 + 동시에 설계 재검토
- **Verification의 첫 번째 역할: "사양대로 구현되었는가?"**
  - Spec에서 **72개 체크 항목**을 식별
  - 각 항목에 대해 설계(Spec)에 맞춰 구현되었는지 확인
  - 결과: ✅ 71개 PASS, ❌ 1개 FAIL (Deviation)
  - 발견된 Deviation: Spec은 `PUT /schema/blog/episode`인데 코드는 `PUT /schema//blog/episode` (이중 슬래시)
- **복잡도가 높을수록 AI가 사양을 놓칠 가능성은 높아진다**
  - jsondb는 단순하지만, 실제 소프트웨어는 훨씬 복잡함
  - 사양이 방대할수록 AI가 모든 세부사항을 정확히 구현할 확률은 낮아짐
  - → **작은 단위로 점진적/반복적 개발** 필요: 큰 Spec 한 번 구현 → 검증 부담 폭증보다 작은 Spec씩 구현 → 검증+피드백 선순환
- **Verification의 두 번째 역할: "설계 자체가 올바른가?"**
  - 사양대로 구현되었는지 검증하면서, **동시에 설계가 올바른지 다시 검토**할 수 있다
  - 복잡도가 높은 경우, 설계 시점에 충분히 고려하지 못해 **부적절한 결정**을 내릴 수 있음
  - "사양대로 구현되지 못했다"는 사실이 **설계의 문제를 드러내는 신호**일 수 있음
  - → 사양대로 안 되면 "AI 잘못"이 아니라 **"설계 자체를 변경해야 하는 상황"** 일 수 있음
- **💡 포인트:** "Verification은 단순한 체크리스트가 아니다. **Spec대로 구현되었는지 확인하는 동시에, 설계 자체가 올바른지 재검토하는 기회다.** "

##### 슬라이드 11: 🤫 Undocumented ASR — AI의 침묵의 기본값
- **Undocumented ASR:** Spec이 명시하지 않은 빈틈을 AI가 자기 마음대로 채운 설계 결정
- **6건 발견:**
  - URL 인코딩 누락
  - 데이터 수정 시 스키마 재검증 생략
  - 등
- **💡 교훈:** Spec이 완벽하더라도, 명시되지 않은 공백은 AI가 임의로 채운다.
- **해결책:** 검증을 통해 발견된 빈틈을 Spec에 추가 → **Spec을 계속 업데이트하는 Living Document 사이클**

#### 2.6 구조는 고정되지 않는다 — Living Cycle (슬라이드 12) — 1.5분

##### 슬라이드 12: 🔄 구조는 고정되지 않는다 — Living Cycle
- **오해: Waterfall** — 구조 설계 → 구현 → 검증 (끝)
- **현실: Living Cycle** — 구조가 개발을 이끌지만, 구조 자체도 계속 진화한다
  ```
         (새 요구사항 / 검증 발견)
                  ↑
  구조 설계(ADR) → Spec → Generation → Verification
      ↑                                    ↓
      └────────── (피드백 반영) ←───────────┘
  ```
- **사이클의 흐름:**
  - 검증에서 문제 발견 → 구조 설계(ADR)로 돌아가 결정 재검토
  - 요구사항 변경 → ADR 변경 → Spec 갱신 → 재생성 → 재검증
  - ADR 결정이 변경될 수 있고, Spec도 함께 갱신된다
- **핵심 개념:**
  - **ADR도 살아있는 문서다** — 요구사항 변화에 따라 결정이 변경될 수 있다
  - **Spec도 살아있는 문서다** — ADR 변경이나 검증 결과를 반영하여 계속 갱신된다
  - **구조가 정해지면 변하지 않는 것이 아니다.** 구조가 개발을 이끌지만, 구조 자체도 지속적으로 개선된다
- **💡 포인트:** "구조 기반 개발은 Waterfall이 아니다. **구조가 개발을 이끌고, 개발 경험이 구조를 개선한다.** "

### 3. 결론 (슬라이드 13~14) — 약 3분

#### 슬라이드 13: 핵심 요약 + 학습 확인
- **4-point 요약:**
  1. **AI의 제안을 검토하라.** 할루시네이션(Hallucination)만 조심할 게 아니다. AI의 제안이 부적절한 이유는 다양하다 — **일반적**이어서, **쉽고 간단**해서, 또는 **품질을 반영하지 못해서**. 사용자의 검토와 승인이 필수다.
  2. **ADR과 Spec은 함께 살아있는 문서다.** 검증에서 문제 발견 또는 요구사항 변경 시, 구조 설계(ADR)로 돌아가 결정을 재검토하고 Spec도 함께 갱신한다. 구조는 한 번 정해지면 변하지 않는 것이 아니다.
  3. **검증이 곧 품질이다.** 72개 검증으로도 모든 걸 잡을 수 없지만, 검증 없이는 그조차 발견할 수 없다.
  4. **Cocrates는 여러분의 구조 설계 역량을 키우는 도구다.** 단순한 코드 생성기가 아니라, 올바르게 AI를 사용하고 설계 능력을 키울 수 있도록 안내한다.
- **학습 확인 (자기 질문):**
  - ❓ Artifact Generation 파이프라인 4단계를 말할 수 있는가?
  - ❓ AI의 제안이 왜 내 요구와 맞지 않는지 3가지 이유를 실제 사례로 설명할 수 있는가?
  - ❓ Undocumented ASR이 무엇이고, 왜 발생하는지 설명할 수 있는가?
  - ❓ 구조 기반 개발이 Waterfall이 아닌 이유를 설명할 수 있는가? ADR과 Spec이 어떻게 함께 진화하는지 말할 수 있는가?
