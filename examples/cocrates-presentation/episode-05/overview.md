# S1: Episode-05 발표 정의

## 제목

**구조 기반 산출물 생성 활동 — 실습**

부제: "jsondb를 만들어줘"라고 요청했더니 일어난 일

## 목적

Cocrates의 **구조 기반 산출물 생성 파이프라인 4단계(ADR → Spec → Generation → Verification)** 를 실제 jsondb 개발 사례를 통해 생생하게 경험한다. 시청자는 AI에게 "개발해 줘"라고 요청하는 것만으로 끝나는 것이 아니라, **구조적 설계 → 결정 기록 → 명세서 → 생성 → 검증 → 구조 재설계**의 순환 과정이 어떻게 이루어지는지 이해하게 된다.

단순한 Waterfall이 아님을 강조한다. 검증에서 문제 발견이나 요구사항 변경 시, 구조 설계(ADR)로 돌아가 결정을 재검토하고 Spec도 함께 갱신하는 **Living Cycle**임을 전달한다. 구조가 정해지면 변하지 않는 것이 아니라, 지속적으로 구조를 갱신하면서 구조 주도로 개발을 이끌어야 한다.

## 청중

- Ep4까지 시청한 개발자 (소크라테스식 학습 파이프라인을 경험함)
- AI에게 "이것 좀 만들어줘"라고 요청해본 경험이 있는 사람
- AI가 생성한 코드를 "되는 것 같으니 넘어간" 경험이 있는 사람
- jsondb 같은 간단한 로컬 저장소를 개발해본 경험이 없어도 무방

## 예상 시간

약 22분 (14슬라이드 × ~1.5분)

## 핵심 메시지

> AI의 제안이 부적절한 이유는 다양하다 — **일반적**이어서, **쉽고 간단**해서, 또는 **품질을 충분히 반영하지 못해서**. 할루시네이션만 조심할 게 아니다. 사용자의 검토와 승인이 필수적이며, Cocrates는 올바르게 AI를 사용하고 구조 설계 역량을 키우는 데 도움을 준다.

구조 기반 생성은 단발성 Waterfall이 아니다. 검증과 요구사항 변경에 따라 **ADR → Spec → Generation → Verification**이 순환하며, 구조가 정해지면 변하지 않는 것이 아니라 지속적으로 갱신된다.

## 성공 기준

시청자가 이 발표를 마친 후:
1. **구조 기반 산출물 생성 파이프라인 4단계**(ADR → Spec → Generation → Verification)를 설명할 수 있다
2. **AI의 제안이 항상 내 요구와 맞지 않는 이유**를 3가지 측면(일반적이다 / 쉽고 간단하다 / 품질을 충분히 반영하지 못한다)으로 실제 사례(저장 모델, 아키텍처, 동시성)를 들어 설명할 수 있다
3. **ADR과 Spec이 함께 살아있는 문서**라는 개념을 이해하고, 검증 결과나 요구사항 변경이 발생하면 구조 설계(ADR)로 돌아가 결정을 재검토하는 **Living Cycle**을 설명할 수 있다
4. **Undocumented ASR(암묵적 설계 결정)** 의 개념을 이해하고, 검증 단계가 왜 중요한지 설명할 수 있다

### 이전 에피소드와의 연결

- **Ep2 (원칙):** "검토되지 않은 산출물은 생성할 가치가 없다" — 이 파이프라인(ADR → Spec → Generation → Verification)이 그 원칙의 구체적 실현이다
- **Ep4 (학습):** 소크라테스식 학습 파이프라인이었다면, Ep5는 구조 기반 산출물 생성 파이프라인이다. 두 파이프라인은 Cocrates의 2가지 기본 활동이다.

## 슬라이드 구성

총 14슬라이드, 도입(2) + 본론(10) + 결론(2) 구조

### 도입 (~3분)
1. **타이틀 페이지** — Ep4 연결 (학습 파이프라인 → 생성 파이프라인)
2. **학습 목표** — 4가지 목표 (파이프라인 설명, AI 제안의 3가지 한계, ADR+Spec Living Cycle, Undocumented ASR)

### 본론 — 파이프라인 경험 + Living Cycle (~16분)
3. **시작: "jsondb 개발해 줘"** — 일반 AI vs Cocrates 응답 비교
4. **ADR ①: 저장 모델 논쟁** — **일반적이다(Generic)** 가 내 요구와 맞지 않았던 사례
5. **ADR ②: 아키텍처 논쟁** — **쉽고 간단하다(Simple)** 가 미래 확장성을 담지 못한 사례
6. **ADR ③: 동시성 모델 논쟁** — **품질을 충분히 반영하지 못한다(Quality-blind)** 사례
7. **ADR 교훈** — AI 제안의 다양한 부적절함 + ADR도 살아있는 문서
8. **Spec 단계** — ADR 결정을 하나의 명세서로 통합
9. **Generation 단계** — Spec만 보고 코드 생성 (7개 Go 파일 + REST + CLI + 테스트)
10. **Verification 단계** — 사양 검증 항목 (72개: 71 pass, 1 fail) + 사양 검증과 동시에 설계 재검토 + 점진적/반복적 개발의 필요성
11. **Undocumented ASR** — 6건의 암묵적 설계 결정
12. **Living Cycle** — 검증/요구사항 변경 → 구조 설계(ADR)로 회귀하는 순환

### 결론 (~3분)
13. **핵심 요약 + 학습 확인** — 4-point 요약 (AI 제안 검토, ADR+Spec Living, 검증=품질, Cocrates 역량)
14. **다음 에피소드 연결** — Ep6: 구조 기반 워크플로우 생성 실습 (스킬 생성)
