# Script — Slide 8 (1분 30초)

## Slide: 🔍 spec-driven-verification — "설계도와 대조하라"

**Time:** ~1분 30초

"네 번째이자 마지막 스킬: **spec-driven-verification**입니다.

검증의 핵심 원칙: **이중 목적 검증** — Deviation(이탈)과 Undocumented ASR, 이 두 가지를 모두 찾습니다.

첫 번째 검증 대상은 **Deviation**입니다.
Spec에는 'PostgreSQL을 써라'고 했는데, 생성물에 MongoDB가 들어가 있습니다.
이것은 명백한 이탈입니다.
그런데 원인을 보면 흥미롭습니다.
설계자의 실수일 수도 있고, 구현 중에 PostgreSQL이 불가능하다는 사실을 발견했을 수도 있습니다.
또는 AI가 문제를 해결하는 과정에서 스스로 MongoDB가 더 낫다고 판단했을 수도 있습니다.

여기서 중요한 점: **사용자가 재검토해야 합니다.**
설계가 잘못된 것일 수도 있고, 구현이 잘못된 것일 수도 있습니다.
AI가 독단적으로 수정하지 않고, 사용자에게 알려서 판단을 받습니다.

두 번째 검증 대상은 **Undocumented ASR**입니다.
Spec에 없었던 내용인데 AI가 생성물에 구현했습니다.
예를 들어, AI가 임의로 하위 섹션을 추가했거나, 스스로 새로운 요구사항이라고 제시한 경우입니다.

Undocumented ASR을 발견하면, 사용자는 3단계로 처리합니다.

첫째, **무시(Ignore)** — AI가 중요하다고 했지만, 사용자 판단에 중요하지 않습니다.
그냥 둡니다. Spec에 포함하지 않습니다.

둘째, **수용(Accept)** — 정말 중요한 요구사항이고 AI의 결정이 타당합니다.
Spec에 포함시킵니다. 다음 생성부터는 공식 요구사항이 됩니다.

셋째, **ADR 검토(Escalate)** — 중요한 요구사항인데, AI의 결정을 따라도 될지 판단이 어렵습니다.
ADR로 보내서 대안을 비교하고 사용자가 직접 결정합니다.

그런데 여기서 끝이 아닙니다.
**검증에는 숨은 목적이 있습니다.**
ASR을 놓친 경험, 잘못된 의사결정 경험을 통해 사용자는 자신의 역량을 개선합니다.
'아, 이걸 미리 결정했어야 했는데...' — 이 깨달음이 다음 프로젝트에서 같은 실수를 방지합니다.

포인트: **검증은 생성물을 검사하는 동시에, 사용자의 다음 설계를 더 나아지게 만드는 피드백 루프입니다.**

자, 이제 4개 스킬을 모두 보았습니다. 이것들이 어떻게 연결되는지 한눈에 정리하겠습니다."
