# 구조 기반 워크플로우 생성 실습 — 설명문/보고서 스킬 만들기

---

지난 편(Ep5)에서 우리는 Cocrates와 함께 `jsondb`를 개발하며 **구조(architecture)의 중요성**을 직접 경험했습니다.

구조는 처음부터 명확하지 않습니다. 모호하기 때문에 ADR로 대안을 검토하고, 결정된 내용을 Spec으로 통합한 뒤, 그 Spec을 기준으로 생성했습니다. 이것이 `spec-driven-generation`입니다. Ep5의 jsondb 실습이 바로 그 과정이었죠.

그런데 한 가지 생각해볼 점이 있습니다.

> ***"산출물 유형별로 '이런 구조로 생성한다'는 워크플로우가 미리 정의되어 있다면, 더 효율적이지 않을까?"***

예를 들어, 보고서를 생성할 때마다 매번 ADR로 구조를 검토하는 대신, **보고서라는 산출물 유형에 최적화된 생성 절차**가 스킬로 정의되어 있다면 어떨까? 개요를 먼저 쓰고, 구조를 설계하고, 섹션별로 작성하는 일련의 과정이 표준화되어 있다면 훨씬 효율적일 것입니다.

Cocrates가 이렇게 명세된 스킬을 사용하면, 사용자는 구조적으로 산출물을 설계하고 생성할 수 있습니다. 즉, **스킬은 구조 기반 생성을 더 효율적으로 만들어주는 워크플로우 템플릿**인 셈입니다.

이번 편에서는 바로 그 워크플로우 — 설명문과 보고서 등의 문서 생성을 위한 `document-authoring` 스킬 — 을 Cocrates와 함께 **직접 설계하고 생성하는 과정**을 따라가보겠습니다.

---

## 실습 시나리오: "설명문/보고서 등을 생성하는 스킬을 생성해줘"

모든 것은 이 한 문장으로 시작되었습니다.

> *"설명문/보고서 등을 생성하는 스킬을 생성해줘."*

Cocrates는 최종 산출물(문서)이 아니라, 그 산출물을 생성하는 **구조적 워크플로우 스킬**을 만들어 달라는 사용자의 의도를 파악합니다.

> **`generating-skill-creation` 발동.**

---


## 1단계 — Kernel: 핵심을 한 문장으로 정의하기

Cocrates가 가장 먼저 한 일은 질문이었습니다. 요청이 너무 일반적이었기 때문에 구체화가 필요했습니다.

> **Cocrates**: "어떤 종류의 설명문/보고서를 주로 생성하게 될까요? 이 스킬을 사용할 대상(독자)은 누구인가요? 생성되는 산출물은 어떤 형태로 저장되나요?"

세 가지 질문은 각각 **무엇을(What), 누가(Who), 어떻게(How)** 에 해당합니다. 스킬의 범위와 성격을 결정짓는 가장 기본적인 축입니다.

> **사용자**: "일반적인 문서. 누구나 독자가 될 수 있음. md 파일."

짧은 답변이었지만 충분히 명확했습니다. Cocrates는 이 응답을 바탕으로 **Kernel 문장**을 정리했습니다.

> **Kernel**: 이 스킬은 Markdown 형식의 설명문/보고서를 검토 가능한 단계를 거쳐 생성하도록 돕는다.

이 한 문장이 앞으로 만들어질 모든 절차의 방향을 결정합니다. Cocrates는 모호한 상태로 절차를 진행하지 않습니다. 사용자의 의도를 명확히 이해했다는 확인을 받을 때까지 질문합니다.

## 2단계 — Frame: 산출물의 구조와 생성 단위 설계하기

Kernel이 확정되자, Cocrates는 다음 단계로 넘어갔습니다. **산출물의 계층 구조를 설계**하는 단계입니다. 이 단계가 없으면 '그냥 글을 쓰는' 수준에 머물게 됩니다.

Cocrates가 제시한 문서의 구조는 다음과 같았습니다.

```
문서(Document)
 ├── 메타정보 (제목, 작성일, 버전, 독자)
 ├── 개요/서론 (문서의 목적, 범위, 핵심 메시지)
 ├── 본문
 │    ├── 섹션 1
 │    │    ├── 서브섹션 1.1
 │    │    └── 서브섹션 1.2
 │    ├── 섹션 2
 │    └── 섹션 3
 ├── 결론/요약
 └── 부록 (참고자료, 용어정의 등)
```

여기에 더해 **생성 단계와 검토 단위**도 함께 제시했습니다.

| 단계 | 산출물 | 검토 단위 |
|------|--------|-----------|
| 1. 개요(Outline) | 문서 메타정보 + 목차 | 사용자 승인 |
| 2. 구조 설계 | 섹션별 핵심 포인트 | 사용자 승인 |
| 3. 본문 생성 | 섹션별 Markdown 내용 | 섹션별 검토 |
| 4. 최종 검토 | 완성된 문서 전체 | 사용자 최종 승인 |

파일 저장 규칙도 이 단계에서 결정되었습니다.

```
docs/{slug}/
├── outline.md              # 개요
├── structure.md            # 구조 설계
├── sections/               # 섹션별 본문
│   ├── 01-introduction.md
│   ├── 02-main-point-1.md
│   └── ...
└── {slug}.md               # 최종 문서
```

그리고 Cocrates는 사용자에게 확인을 요청했습니다.

> **Cocrates**: "이 구조에 대해 어떻게 생각하시나요? 섹션 단위로 검토하는 것이 좋을지, 한 번에 작성하는 것을 선호하시나요?"

> **사용자**: "일반적으로는 섹션 단위로 검토하고 작성하는 것이 좋지."

이 응답으로 세 가지가 확정되었습니다.
- **검토 단위**: 섹션별
- **저장 형식**: Markdown 파일
- **생성 계층**: 개요 → 구조 → 본문

## 3단계 — Outline: 구체적인 생성 절차 설계하기

Frame이 승인되자 Cocrates는 각 단계를 더 상세하게 설계했습니다. 여기서 중요한 것은 각 단계의 **입력(Input), 출력(Output), 승인 조건(Approval Condition)** 을 명확히 하는 것입니다.

Cocrates가 제안한 5단계 절차는 다음과 같습니다.

| 단계 | 목적 | 입력 | 출력 | 승인 조건 |
|------|------|------|------|-----------|
| **P1. 개요 설계** | 문서의 목적·범위·독자·핵심 메시지 정의 | 사용자 요청 | `outline.md` | 사용자 승인 |
| **P2. 구조 설계** | 각 섹션의 핵심 포인트와 흐름 정의 | outline.md | `structure.md` | 사용자 승인 |
| **P3. 섹션 생성** | 섹션별 본문을 순차적으로 작성 | structure.md | `sections/NN-*.md` | 섹션별 승인 |
| **P4. 문서 통합** | 모든 섹션을 하나의 문서로 병합 | sections/*.md | `{slug}.md` | 자동 |
| **P5. 최종 검토** | 완성된 문서 전체 검토 | 최종 문서 | 검토 의견 | 사용자 최종 승인 |

### 이 단계에서 가장 중요한 대화

Cocrates가 섹션 생성 순서에 대해 자연스러운 질문을 던졌습니다.

> **Cocrates**: "섹션 생성 순서는 서론→본론→결론 순으로 자연스러울까요?"

이것은 Cocrates가 가지고 있던 **무의식적 가정**이었습니다. '보고서'라고 하면 일반적으로 서론-본론-결론을 떠올리기 때문입니다.

하지만 사용자의 답변은 전혀 달랐습니다.

> **사용자**: "항상 그런 순서는 아니다. 기승전결일 수도 있고, 내용이나 목적에 따라 적합한 구성을 채택하는 것이 좋겠음."

이 응답은 단순한 순서 조정 이상의 의미를 가졌습니다. 그것은 **스킬의 핵심 설계 원칙 자체를 변화시켰습니다.**

Cocrates는 즉시 이 피드백을 수용했습니다.

> **Cocrates**: "좋은 지적입니다. 문서의 목적과 내용에 따라 가장 적합한 구성을 선택할 수 있어야 하죠. 이를 구조 설계 단계(P2)에서 사용자와 함께 결정하도록 반영하겠습니다."

여기서 중요한 점은 Cocrates가 사용자의 의견을 단순히 '듣고 수용'하는 것을 넘어, 그것을 **스킬의 구조적 원칙으로 승격**시켰다는 것입니다. 이 한마디가 나중에 '맞춤형 문서 구성'이라는 핵심 원칙이 됩니다.

## 4단계 — Spec: 스킬 명세 규칙 확정하기

Outline이 승인되자, Cocrates는 지금까지의 모든 논의를 종합하여 **Spec(명세)** 으로 정리했습니다.

Spec에 포함된 핵심 내용은 다음과 같습니다.

### 스킬 정보
- **이름**: `document-authoring`
- **핵심 원칙 5가지**:
  1. **Snowflake 점진적 구체화**: 개요 → 구조 → 섹션 생성 → 통합 → 최종 검토
  2. **사용자 승인 게이트**: 각 단계(P1~P5)마다 사용자가 검토하고 승인한 후에만 진행
  3. **맞춤형 문서 구성** ⭐ — 문서의 목적과 독자, 전달할 메시지에 따라 가장 적합한 구성 방식을 선택
  4. **섹션 단위 작성 및 검토**
  5. **문서의 목적 명확화**

### 맞춤형 구성 방식 선택지
이 부분이 바로 앞서 사용자의 피드백이 반영된 핵심 설계입니다.

| 문서 목적 | 권장 구성 방식 |
|-----------|---------------|
| 설득/제안 | 서론-본론-결론 |
| 스토리/사례 중심 | 기승전결 |
| 문제 해결 | 문제-원인-해결-제안 |
| 비교/분석 | 기준-대안-평가-결론 |
| 절차/방법 | 단계별 순서 |
| 기타 | 사용자 정의 구성 |

### 작업 경로와 파일 구조
```
docs/{slug}/
  ├── outline.md              # P1
  ├── structure.md            # P2 (구성 방식 포함)
  ├── sections/               # P3
  │   ├── 01-introduction.md
  │   ├── 02-main-point-1.md
  │   └── ...
  └── {slug}.md               # P4
```

### 금지 사항 (7가지)
- P1(개요)이나 P2(구조) 사용자 승인 없이 본문 생성(P3) 시작
- 섹션의 사용자 승인 없이 다음 섹션으로 넘어가기
- P5(최종 검토) 사용자 승인 없이 문서 생성 완료 처리
- 중간 산출물을 파일로 저장하지 않고 채팅에만 남기기
- 문서 생성 중 다른 유형의 산출물 생성
- 사용자가 승인하지 않은 내용을 문서에 포함시키기
- 전용 스킬이 더 적합한 문서 유형에 이 스킬 사용

### 완료 조건 (5가지)
- 최종 문서 파일이 생성됨
- P1~P5의 모든 단계를 거쳐 사용자가 최종 승인함
- 모든 중간 산출물이 파일로 저장됨
- 문서의 목적, 독자, 핵심 메시지가 명확히 반영됨
- 사용자가 문서의 내용과 구조를 설명할 수 있음

그리고 Cocrates는 다시 한 번 확인을 요청했습니다.

> **Cocrates**: "이 Spec에 대해 검토해 주시겠어요? 추가하거나 수정할 부분이 있나요?"

> **사용자**: "좋아. 스킬을 생성해줘."

**Cocrates는 Spec을 혼자서 확정하지 않습니다.** 반드시 사용자의 검토와 승인을 거칩니다. Spec은 Cocrates와 사용자의 **공동 설계물**입니다.

## 5~6단계 — Skill 생성 및 Verification

승인된 Spec을 바탕으로 Cocrates는 `.opencode/skills/document-authoring/SKILL.md`를 생성했습니다.

### 생성된 SKILL.md의 주요 구성

| 구성 요소 | 내용 |
|----------|------|
| **5가지 핵심 원칙** | Snowflake 점진적 구체화, 승인 게이트, 맞춤형 문서 구성, 섹션 단위 검토, 목적 명확화 |
| **사용 시점/비사용 시점** | 일반 문서 요청 시 사용, 전용 스킬이 더 적합하면 사용하지 않음 |
| **5단계 절차(P1~P5)** | 각 단계마다 목적·절차·검토 질문·승인 조건이 8~15줄로 상세히 명시 |
| **작업 위치와 파일 트리** | `docs/{slug}/` 아래 outline → structure → sections → 최종 문서 |
| **중간 산출물 정의** | outline.md, structure.md, sections/NN-*.md 각각의 템플릿과 예시 |
| **대화 규칙 6개** | 단계 명시, 검토 질문 순차 제시, '일단 만들어줘'에도 구조 승인 우선 등 |
| **금지 사항 7개 + 완료 조건 5개** | 구체적이고 검증 가능한 형태로 명시 |

### 마지막 단계 — Verification

Cocrates는 스킬을 생성한 후, 그것이 제대로 작동하는 **하네스**인지 검증했습니다. 검증 체크리스트는 다음과 같습니다.

| 검증 항목 | 결과 |
|----------|------|
| description이 무엇을/언제 사용하는지 명확한가 | ✅ 통과 |
| 최종 산출물보다 중간 구조 산출물을 먼저 만들도록 지시하는가 | ✅ 통과 |
| 사용자 승인 없이 다음 단계로 넘어가지 않는 게이트가 있는가 | ✅ 통과 |
| 산출물의 계층 구조와 파일 저장 방식이 명확한가 | ✅ 통과 |
| 완료 조건과 금지 사항이 구체적인가 | ✅ 통과 |
| 기존 스킬과 용어가 충돌하지 않는가 | ✅ 통과 |
| 같은 유형의 요청에 재사용할 수 있는가 | ✅ 통과 |

**모든 항목 통과 — 스킬 생성 완료.**

## 이 스킬이 갖춘 5가지 핵심 설계 요소

생성된 `document-authoring` 스킬을 특징짓는 다섯 가지 핵심 요소를 정리해보겠습니다.

### ① Snowflake 5단계 (P1~P5)

```
P1(개요 설계) → P2(구조 설계) → P3(섹션 생성) → P4(문서 통합) → P5(최종 검토)
```

각 단계는 명확한 입력, 출력, 승인 조건을 가집니다. P3에서는 **섹션별로 사용자 검토**를 받으며 순차적으로 진행합니다. 한 번에 모든 내용을 생성하는 것이 아니라, 검증 가능한 단위로 쪼개서 진행합니다.

### ② 맞춤형 문서 구성 (Custom Document Composition)

문서의 목적에 따라 6가지 구성 방식 중 선택할 수 있습니다. 이 원칙은 사용자의 피드백 **"항상 서론→본론→결론은 아니다"** 에서 직접 비롯되었습니다.

Cocrates는 초기에 서론-본론-결론이라는 일반적인 가정을 가지고 설계를 시작했지만, 사용자의 지적으로 인해 훨씬 더 유연하고 강력한 원칙을 갖게 되었습니다. 이것은 AI가 무의식적으로 가지고 있는 '침묵의 기본값'을 사용자가 직접 깨트린 사례입니다.

### ③ 사용자 승인 게이트

```
P1: 사용자 승인 필요
P2: 사용자 승인 필요 (여기서 구성 방식 결정)
P3: 섹션별 사용자 승인 필요
P4: 자동 진행 (병합만 수행)
P5: 사용자 최종 승인 필요
```

4개의 승인 게이트가 문서의 품질을 보장합니다. 사용자는 '동의'와 '거절'만 하는 것이 아니라, 각 게이트에서 **검토 질문**을 통해 문서의 방향성을 함께 조정합니다.

### ④ 중간 산출물 파일 관리

모든 중간 산출물은 `docs/{slug}/` 디렉토리 하위에 파일로 저장됩니다. 채팅에만 내용을 남기지 않습니다. 이는 다음과 같은 장점이 있습니다.

- 사용자가 중간 단계를 다시 검토할 수 있음
- 수정이 필요할 때 이전 단계로 쉽게 돌아갈 수 있음
- 협업 시 산출물을 공유할 수 있음

### ⑤ 명확한 금지 사항과 완료 조건

7가지 금지 사항은 **하지 말아야 할 것**을 명확히 정의하여 Cocrates가 잘못된 방향으로 가는 것을 방지합니다. 5가지 완료 조건은 **언제 이 스킬의 작업이 끝났는지**를 명확히 합니다.

## 스킬 생성 이후 — 새로운 스킬로 문서 생성하기

이제 `document-authoring` 스킬이 등록되었습니다. 이후 "~에 관한 보고서를 생성해줘"라는 요청이 들어오면 Cocrates는 이 스킬을 선택합니다.

Cocrates는 더 이상 "보고서 작성해줘"라는 요청을 무작정 수행하지 않습니다. 대신 `document-authoring` 스킬의 절차에 따라 행동합니다.

1. **P1 — 개요 설계**: "어떤 주제의 보고서인가요? 누가 읽나요? 핵심 메시지는 무엇인가요?"
2. **P2 — 구조 설계**: "이 보고서는 어떤 구성이 적합할까요? 서론-본론-결론? 문제-원인-해결? 기승전결?"
3. **P3 — 섹션 생성**: "첫 번째 섹션을 작성했습니다. 검토해 주세요."
4. **P4 — 통합**: (자동)
5. **P5 — 최종 검토**: "전체 문서를 검토해 주세요. 최종 승인하시겠습니까?"

사용자는 더 이상 아무렇게나 생성된 문서를 받지 않습니다. 대신 **자신이 설계에 참여하고, 모든 단계를 검토하고, 최종 승인한 문서**를 받게 됩니다.

이것이 Cocrates의 약속입니다. **"검토되지 않은 산출물은 생성할 가치가 없다."**

---

## 이 실습의 교훈 — 스킬 생성이란 '함께 설계하는 과정'이다

이 실습이 주는 가장 중요한 교훈은 무엇일까요?

1. **Cocrates는 '스킬 생성'과 '산출물 생성'을 정확히 구분한다.** 사용자가 "스킬을 생성해줘"라고 요청하면, Cocrates는 `generating-skill-creation`을 발동시켜 산출물(문서)이 아니라 **절차(SKILL.md)** 를 만든다. 이 구분이 모호하면, 그냥 문서 하나를 만들어주고 끝나는 일회성 작업이 된다.

2. **사용자의 피드백이 스킬의 핵심 원칙이 된다.** 이 실습에서 가장 극적인 순간은 사용자가 "항상 서론→본론→결론은 아니다"라고 말했을 때였습니다. Cocrates는 이 의견을 단순히 수용하는 것을 넘어, **'맞춤형 문서 구성'** 이라는 스킬의 핵심 원칙으로 승격시켰습니다. 이 스킬을 사용하는 모든 미래의 사용자들은 이 원칙의 혜택을 받게 됩니다. 이것이 '함께 설계한다'는 의미입니다.

3. **스킬도 검증된다.** AI가 생성한 모든 것은 검증되어야 합니다. 스킬도 예외가 아닙니다. Cocrates는 생성된 SKILL.md가 7가지 검증 기준을 통과했는지 확인한 후에야 스킬 생성을 완료했습니다.

4. **한 번 만든 스킬은 재사용된다.** 이 스킬이 등록된 이후, 같은 유형의 요청(문서 생성)은 Cocrates가 바로 이 스킬을 사용하여 처리합니다. `generating-skill-creation`이 다시 발동할 필요가 없습니다. 스킬은 **일회성이 아니라 축적되는 자산**입니다.

## 당신의 차례입니다

이제 `document-authoring` 스킬이 Cocrates Harness에 추가되었습니다.

"~에 관한 보고서를 생성해줘"라고 요청해보세요. 업무 보고서, 학습 정리 문서, 분석 보고서, 제안서 — 무엇이든 좋습니다.

Cocrates는 여러분과 함께:
1. 문서의 목적과 독자를 명확히 하고
2. 목적에 가장 적합한 구성을 선택하며
3. 섹션별로 검토하고 승인받으며
4. 최종 검토를 거쳐 완성할 것입니다

단순히 '생성' 버튼을 누르는 것이 아니라, **여러분이 설계하고 검토하고 승인한 문서**를 받아보는 경험을 직접 해보시기 바랍니다.
