# 소크라테스식 학습 활동 - 스킬

---

지난 편에서 우리는 Cocrates의 Learning 파이프라인이 왜 세 단계(Education → Knowledge Capture → Reflection)로 구성되어 있는지, 그 개념적 토대를 살펴봤습니다. 소크라테스 산파술, 블룸의 교육분류학, 근접발달영역 — 이 세 철학이 어떻게 Cocrates의 학습 활동을 지탱하는지 알게 되었습니다.

이제 그 철학이 **실제 시스템에서는 어떻게 구현되어 있는지** 들여다볼 차례입니다.

`.opencode/skills/` 디렉토리 아래에는 education, knowledge-capture, reflection — 세 개의 SKILL.md 파일이 있습니다. 각 파일은 단순한 프롬프트가 아닌, 정교하게 설계된 워크플로우 명세입니다.

하나씩 열어보겠습니다.

---

## Education — 소크라테스형 1:1 학습 코치

### 가장 강력한 규칙: No Spoon-feeding

education 스킬의 첫 번째 규칙은 이것입니다.

> **절대 완전한 정답이나 전체 해결책을 한 번에 주지 않는다.**

사용자가 "DIP 원칙에 대해 알려줘"라고 요청해도, Cocrates는 개념 설명을 길게 늘어놓지 않습니다. 핵심 원리를 **1~3문장, 일상적인 비유**로만 전달합니다. 전체 응답의 20%를 넘지 않는 선에서 말이죠.

그리고 항상 **불완전한 상태(incomplete state)** 로 대화를 멈춥니다. 결함이 있는 예시, 열린 간극을 남겨두고 — 사용자가 직접 채우도록 기다립니다.

### 세 블록 형식

모든 education 응답은 세 블록으로 구성됩니다:

```markdown
### 💡 [Concept Briefing]
(핵심 원리를 1~3문장, 일상적인 비유로. 전체의 20% 이하.)

### 💻 [Thought Lab]
(결함이 있는 실용적 예시나 확장된 시나리오.)

### 🔥 [MISSION]
(사용자가 다음 턴에 생각하고 답변해야 할 정확히 하나의 과제.)
```

실제 대화를 예로 들어볼까요. 사용자가 "의존성 역전 원칙이 뭐야?"라고 묻는다면:

**Cocrates:**

### 💡 [Concept Briefing]
DIP는 전원 플러그와 소켓 표준과 같습니다. 고수준 모듈이 저수준 세부사항에 직결되지 않고, 둘 다 공유된 인터페이스에 의존해야 합니다.

### 💻 [Thought Lab]
```typescript
class OrderService {
  constructor(private db: MySQLDatabase) {}
  // 문제: 고수준이 구체적 저수준에 직접 의존하고 있음
  createOrder() { /* ... */ }
}
```

### 🔥 [MISSION]
`MySQLDatabase`를 `PostgresDatabase`로 바꾸면 무엇이 깨질까요? DIP를 위반하는 의존 방향을 하나 찾아 자신의 말로 설명해 보세요.

---

이것이 education 스킬의 전부입니다. 개념 브리핑은 아주 짧게, 생각의 실험실은 구체적인 코드나 시나리오로, 그리고 미션으로 사용자가 직접 생각하고 답변하게 만듭니다.

정답을 주지 않습니다. **다음 미션의 답을 미리 알려주지도 않습니다.** 사용자의 답변을 기다립니다.

### Bloom's 2D Matrix — 단순한 분류학이 아니다

education 스킬이 사용하는 Bloom's Taxonomy는 **1차원 사다리가 아닌 2차원 매트릭스**입니다.

| | **사실적 (Factual)** | **개념적 (Conceptual)** | **절차적 (Procedural)** | **메타인지 (Metacognitive)** |
|---|---|---|---|---|
| **기억→이해** | 단어·용어 정의 | 개념 관계 이해 | 절차 파악 | 전략 인식 |
| **적용→분석** | 사실 기반 분석 | 개념 적용·비교 | 절차 수행·비교 | 전략 평가 |
| **평가→창조** | 사실 판단 | 새 개념 설계 | 새 절차 설계 | 메타 시스템 설계 |

Cocrates는 **두 축을 동시에 올리지 않습니다.** 같은 지식 차원에서 인지 과정을 한 단계 높이거나, 같은 인지 과정에서 지식 차원을 한 단계 넓힙니다.

그리고 사용자의 상태에 따라 **Push(하→상 단계별 안내)와 Pull(상위 도전 먼저 제시)** 전략을 선택합니다. 초보자나 인지적 붕괴(cognitive collapse)가 보이면 Push로 전환하고, 사전 지식이 있는 사용자에게는 Pull로 더 높은 도전을 먼저 제시합니다.

### 언제 종료되는가

education 스킬은 세 가지 조건이 충족되면 종료됩니다:

1. 사용자가 핵심 개념을 **자신의 말로 설명**할 수 있을 때
2. 현재 단계의 미션(MISSION)이 확인되었을 때
3. 사용자가 자신의 **이해 공백(Gap)** 을 명시적으로 인식했을 때

종료 직후 Cocrates는 자연스럽게 knowledge-capture를 제안합니다: *"이 내용을 KB에 정리해 드릴까요?"*

---

## Knowledge Capture — 배운 것을 기록하다

Education 단계에서 학습한 내용은 기억에 의존해서는 안 됩니다. knowledge-capture 스킬은 학습한 내용을 `kb/` 폴더에 **회상(recall)용 핵심만** 저장합니다.

### 상세 설명이 아니라, 회상용 핵심

가장 중요한 원칙은 이것입니다: **상세한 강의 노트, 긴 설명, 전체 코드를 저장하지 않는다.**

하나의 bullet은 하나의 기억 단위 — 1~2문장을 넘지 않습니다.

실제 저장되는 파일의 형식을 보겠습니다:

```markdown
# Dependency Inversion Principle

## One-line Definition
고수준 모듈이 저수준 구현에 직접 의존하지 않고, 둘 다 추상화(인터페이스)에 의존해야 한다는 원칙.

## Key Points
- 고수준 모듈은 저수준 구현에 직접 의존해서는 안 된다
- 인터페이스(추상화)에 의존하고, 구현체는 주입(Injection)을 통해 받는다
- DIP는 단순히 "인터페이스를 쓰라"가 아니라, **의존의 방향**이 핵심이다

## Wrong Assumptions / Gaps
- ~~DIP = 무조건 새로운 인터페이스 파일을 만드는 것~~ (X)
  → 기존 코드와의 경계에서는 Adapter를 활용해 연결하는 것도 유효한 방법이다

## Tags
`solid`, `dependency-inversion`, `architecture`
```

**핵심은 `Wrong Assumptions / Gaps` 섹션입니다.**

> 무지(ignorance)의 기록 — 이것이 Knowledge Capture의 핵심입니다. 내가 무엇을 틀렸는지를 기록하는 것이 가장 강력한 학습 도구입니다.

### 병합 전략 — 중복을 허용하지 않는다

knowledge-capture 스킬은 저장 전에 항상 `kb/` 디렉토리를 검색합니다. 같은 주제의 파일이 이미 존재하면 **새 파일을 만들지 않고, 기존 파일에 보충(merge)합니다.**

- `dip-principle.md` ↔ `dependency-inversion-principle.md` → 같은 주제, 병합
- 기존 KB의 내용은 절대 덮어쓰지 않고, 새 인사이트만 추가
- 날짜와 추가 내용을 `## Update History`에 기록

사용자가 배운 내용이 쌓일수록 KB는 단순한 노트가 아니라 **개인 지식 베이스**로 성장합니다. 그리고 이 KB는 다음 단계 — Reflection의 핵심 자료가 됩니다.

## Reflection — 진짜 아는 것과 그냥 아는 척하는 것

Learning 파이프라인의 마지막이자 가장 중요한 단계입니다.

### 면접관(Interviewer) 페르소나

reflection 스킬의 가장 독특한 점은 **페르소나**입니다. Cocrates는 여기서 **코치도, 선생님도 아닌 면접관**이 됩니다.

| | 가르치기 (Teaching) | 반성 평가 (Reflection) |
|---|---|---|
| **역할** | 지식을 쌓는 코치 | 이해를 드러내는 면접관 |
| **목표** | 사용자가 새 통찰에 도달하도록 안내 | 사용자가 이미 알고 있는 것과 모르는 것을 정확히 파악 |
| **성공** | 사용자가 새 깨달음을 얻음 | 사용자와 면접관 모두 지식의 지도를 봄 |

Reflection의 목표는 점수를 매기거나 순위를 정하는 것이 아닙니다. 사용자가 스스로 이렇게 말할 수 있게 하는 것입니다:

> *"I know that I know nothing."*

### KB를 평가 기준으로 사용

Reflection은 저장된 KB 파일을 **평가 기준(rubric)** 으로 사용합니다. KB의 `## Key Points` 항목이 체크리스트가 되고, `## Wrong Assumptions / Gaps`는 사용자가 실제로 그 간극을 메웠는지 확인하는 도구가 됩니다.

단, 중요한 차이가 있습니다: **KB는 정답지(answer key)가 아닙니다.** 사용자의 현재 이해가 KB와 다르지만 논리적으로 일관된다면, KB 업데이트를 제안하는 것이 먼저입니다.

### 질문의 깊이 — 단순 암기를 확인하지 않는다

reflection 스킬이 던지는 질문은 정의를 묻는 것이 아닙니다:

- *"이 원칙을 **다른 도메인의 예시**로 설명해 보세요."*
- *"이 코드는 어디서 **원칙을 위반**하고 있나요?"*
- *"**반대로 한다면** 무엇이 깨질까요?"*
- *"KB에 이 Gap을 기록했는데, 오늘도 같은 실수를 할까요?"*

면접관은 내부적으로 세 단계로 판단합니다:

| 수준 | 신호 |
|------|------|
| **Solid** | 자신의 말로 설명, 올바른 적용, 반례에서도 유지 |
| **Partial** | 방향은 맞지만 경계가 흐릿하거나 용어 혼동 |
| **Gap** | 암기만 가능, 모순 발생, 침묵 |

그리고 최종 산출물은 다음과 같은 형식으로 제공됩니다:

```markdown
## ✅ What you know well
- DIP의 의존 방향: 다른 도메인(결제 시스템) 예시로 올바르게 설명함
- 인터페이스 분리: Adapter 패턴과의 관계를 자신의 말로 서술

## ⚠️ What was missing or shaky (surfaced this review)
- DIP와 Interface Segregation의 경계: 두 원칙이 겹치는 부분에서 혼동 (재확인 필요)
```

점수나 등급이 아닙니다. **관찰(observation)과 발견(discovery)** 입니다.

### Gap이 발견되면?

여기가 중요합니다. reflection 스킬은 교육 모드로 전환하지 않습니다. Gap이 발견되면:

1. 더 좁은 각도로 한 번 더 질문을 던져 봅니다 (재확인)
2. 그것도 안 되면 **⚠️ 항목으로 기록**
3. 별도의 learning session을 제안합니다 — *"이 부분은 교육 세션을 따로 진행하시겠어요?"*

Reflection은 교육이 아닌 **평가**입니다. 사용자가 동의하지 않는 한 교육 모드로 전환하지 않습니다.

## 세 스킬의 연결과 순환

이 세 스킬은 독립적으로 존재하지 않습니다. 하나의 완전한 학습 사이클을 만듭니다.

```
Education (탐구) → Knowledge Capture (기록) → Reflection (검증) → (필요시) Education
```

각 스킬의 **종료 조건이 다음 스킬의 시작 조건**과 연결되어 있습니다:

1. **Education**이 사용자의 이해를 확인하면 → **Knowledge Capture** 제안: *"KB에 정리해 드릴까요?"*
2. **Knowledge Capture**가 저장을 완료하면 → **Reflection** 제안: *"이 내용을 바탕으로 이해도를 확인해 볼까요?"*
3. **Reflection**에서 Gap이 발견되면 → 다시 **Education** 제안: *"이 부분을 더 학습하시겠어요?"*

이 순환은 사용자가 **능동적으로 참여할 때만** 작동합니다. Cocrates가 제안은 하지만, 실제로 움직이는 것은 사용자입니다.

---

지금까지 Learning 파이프라인을 구성하는 세 스킬의 내부를 살펴봤습니다. Education은 질문과 미션으로 사용자가 스스로 발견하게 하고, Knowledge Capture는 그 발견을 기록으로 남기게 하며, Reflection은 그 기록을 바탕으로 진짜 이해를 확인합니다.

이 세 스킬은 `.opencode/skills/education/SKILL.md`, `.opencode/skills/knowledge-capture/SKILL.md`, `.opencode/skills/reflection/SKILL.md` — 각각의 파일로 존재합니다. 여러분이 직접 열어보면 지금 설명드린 워크플로우가 그대로 명세되어 있는 것을 확인할 수 있습니다.

다음 편에서는 Cocrates의 두 번째 핵심 활동 — **구조 기반 산출물 생성 활동**으로 넘어갑니다. ADR부터 Spec, 생성, 검증까지. 산출물을 만들 때 Cocrates가 왜 구조(architecture)부터 묻는지, 그 이유를 알아보겠습니다.
