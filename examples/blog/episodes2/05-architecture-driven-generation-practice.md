# 구조 기반 산출물 생성 활동

---

지난 편(Ep4)에서 우리는 Cocrates의 Learning 파이프라인을 직접 경험했습니다. Cocrates가 단순히 정보를 전달하는 대신, 질문과 미션으로 스스로 발견하도록 이끈다는 것을 체감했을 것입니다.

이번에는 Cocrates의 또 다른 핵심 기능인 **산출물 생성(Generation)** 을 실습합니다. Cocrates는 요청을 받으면 무작정 생성하지 않습니다. 먼저 ADR(Architecture Decision Record)로 대안을 검토하고, 결정된 내용을 Spec으로 통합한 뒤, 그 Spec에 따라 생성합니다. 그리고 생성된 결과가 Spec과 일치하는지 검증(Verification)까지 수행합니다.

이 모든 과정이 실제로 어떻게 진행될까요?

**실제 대화를 기반으로 한 실습**을 통해, Cocrates와 함께 jsondb라는 로컬 JSON 문서 저장소를 개발한 전체 과정을 따라가보겠습니다.

---

## 실습 시나리오: "jsondb를 개발해줘"

모든 것은 이 한 문장으로 시작했습니다.

> *"cocrates/jsondb에 jsondb를 개발해줘."*

평소 같았으면 AI는 바로 "물론입니다!"라며 코드를 쏟아냈을 것입니다. 하지만 Cocrates는 다르게 행동했습니다.

> *"이 jsondb의 용도는 무엇인가요?"*

질문이 돌아왔습니다. Cocrates는 바로 생성하지 않았습니다. 대신 용도, 기능, 기술 스택, 프로젝트와의 관계를 물으며 요구사항을 명확히 했습니다. 목적이 드러나자 Cocrates는 이렇게 선언했습니다.

> *"아키텍처를 설계하고, 검토받고, 승인받은 후에 생성하겠습니다. 그 전에, 무엇을 만드는지 먼저 이해해야 합니다."*

이것이 Cocrates가 생성 요청을 받았을 때 실제로 하는 첫 번째 일입니다. 생성(Do) 이전에 계획(Plan)을 먼저 세우는 것 — 가장 기본적이면서도 가장 중요한 원칙입니다.

## ADR 단계 — AI는 '쉬운 방법'을 제시하지만, 사용자가 결정한다

Cocrates가 대안을 제시할 때마다 한 가지 패턴이 반복되었습니다. **가장 쉽고 간단한 방법을 추천**하는 경향이 있었습니다. 하지만 사용자는 그 추천을 무비판적으로 받아들이지 않았습니다. 오히려 자신의 상황에 맞게 질문하고, 대안을 비교하고, 때로는 Cocrates의 추천을 **뒤집는 결정**을 내렸습니다.

이 실습에서 우리는 세 가지 ADR에 주목합니다. 각각은 AI가 놓치기 쉬운 **다른 종류의 통찰**을 대표합니다.

### ADR 1 — 저장 모델: AI의 무의식적 가정

Cocrates가 가장 먼저 던진 질문은 저장 모델에 관한 것이었습니다. AI답게 세 가지 옵션을 제시했습니다.

> 1. **Collection-Document 구조** (NoSQL 스타일, 가장 일반적)
> 2. **Path-Addressable 구조** (파일 경로 직접 매핑)
> 3. **Key-Value Store** (플랫 K/V)

그리고 이렇게 추천했습니다. *"가장 일반적인 패턴은 Collection-Document 구조입니다. 대부분의 문서 DB가 이 방식을 따릅니다."*

하지만 사용자의 반응은 예상과 달랐습니다.

> *"지금 구조는 collection - set of documents의 db인데, 오히려 path/filename.json 으로 저장되는 것이 필요한데..."*

사용자가 찾은 방향은 **Path-Addressable**이었습니다. 사용자가 지정한 path가 그대로 파일 경로가 되는 모델 — `Set("episode/e1", data)`를 호출하면 `{dbroot}/episode/e1.json`에 저장되는 방식입니다.

| | AI의 첫 제안 | 사용자의 결정 |
|---|---|---|
| **모델** | Collection-Document (NoSQL) | **Path-Addressable** |
| **파일 구조** | 추상화된 스토리지 엔진 | 사용자 path = 파일 시스템 경로 |
| **이유** | 가장 일반적인 패턴 | AI 문서 저장의 실제 사용 방식 |

**교훈**: AI가 익숙한 패턴(Collection-Document)을 가정하고 설계를 시작했지만, 실제 필요는 완전히 달랐습니다. 사용자가 자신의 요구사항을 명확히 인지하고 방향을 정정했습니다. AI는 당연하게 여기는 가정이 있을 수 있다는 것을 항상 염두에 두어야 합니다.

---

### ADR 2 — 사용 맥락의 차이: Library-Only → Client-Server

두 번째 ADR은 더 극적인 반전을 가져왔습니다. Cocrates의 초기 아키텍처 설계는 이러했습니다.

> **Library-Only + CLI**: `sync.RWMutex` 하나로 전체 DB의 동시성을 보호하는 단순한 라이브러리

Cocrates는 "일단 라이브러리로 시작하고, 필요하면 나중에 서버를 추가하자"는 접근을 추천했습니다. 가장 구현이 간단하고 일반적인 패턴이기 때문입니다.

하지만 사용자가 던진 한마디가 모든 것을 바꿨습니다.

> *"동시에 여러 프로세스에서 요청하는 경우에 문제가 되겠네..."*

사용자는 자신이 상상하는 사용 시나리오를 말하기 시작했습니다. **복수의 AI 에이전트가 같은 jsondb에 접근**하는 상황 — 각자의 컨텍스트를 유지하면서 동시에 읽고 쓰는 시나리오였습니다.

> *"client-server 구조로. REST API를 제공하는 서버로 만들고, 엔진은 thread-safe하게."*

Cocrates는 이 제안을 검토하고 이렇게 답했습니다.

> *"큰 변화입니다. 하지만 사용자의 제안이 실제로 더 나은 아키텍처임을 인정하고 ADR을 재작성하겠습니다."*

최종 결정은 **Library + REST Server + CLI** 구조였습니다. jsondb 코어 라이브러리 위에 REST API 서버(`jsondbd`)가 있고, CLI(`jsondb`)는 서버를 통해 데이터에 접근하는 방식입니다.

| | AI의 첫 제안 | 사용자의 결정 |
|---|---|---|
| **아키텍처** | Library-Only + CLI | **Library + REST Server + CLI** |
| **동시성** | 단일 프로세스 가정 | 복수 프로세스/에이전트 지원 |
| **사용 패턴** | Go 코드에서 직접 호출 | HTTP API로 원격 접근 |
| **이유** | 구현이 가장 간단함 | 실제 사용 시나리오(복수 AI 에이전트) |

**교훈**: AI는 단일 프로세스 사용을 가정하고 가장 단순한 방법(RWMutex)을 제시했습니다. 하지만 사용자는 "복수의 AI 에이전트가 같은 DB를 사용한다"는 시나리오를 떠올렸고, 이 관점의 차이가 아키텍처 전체를 뒤바꿨습니다. **사용자가 자신의 사용 맥락을 가장 잘 압니다.** AI는 그 맥락을 모르기 때문에 질문해야 하고, 사용자는 그 질문에 답함으로써 자신이 무엇을 원하는지 명확히 인식하게 됩니다.

---

### ADR 3 — 성능을 고려한 동시성 모델: DB-level → Per-File

서버 아키텍처가 결정된 후, Cocrates는 동시성 모델을 다시 검토했습니다. 처음에는 이런 설계를 제안했습니다.

> **DB-level RWMutex**: 하나의 `sync.RWMutex`로 전체 DB를 보호. 읽기는 공유, 쓰기는 독점.

이것은 코드 3줄이면 끝나는 가장 간단한 구현입니다. 하지만 사용자가 다시 질문했습니다.

> *"복수 프로세스에서 복수 요청에 대해서 DB 전체를 lock하는 것은 너무 큰 비효율인 것 같아. Per-File Mutex Map에 대해 추가 검토해줘."*

사용자의 직관은 옳았습니다. 서버 환경에서는 여러 요청이 동시에 들어옵니다. 이때 서로 다른 파일에 대한 요청이라면 굳이 Lock으로 막을 이유가 없습니다. 같은 파일에 대한 요청만 동기화하면 충분합니다.

Cocrates는 Per-File RWMutex Map을 재설계했습니다.

> **`sync.Map[string]*fileLock`**: 각 파일에 `sync.RWMutex` 할당. 같은 파일은 Read 공유 / Write 독점, 다른 파일은 완전 동시 처리. `List()`는 Lock-free (filesystem walk, best-effort).

사용자가 이 설계를 승인했습니다.

| | AI의 첫 접근 | 사용자의 결정 |
|---|---|---|
| **Lock 단위** | DB 전체 (단일 RWMutex) | **파일별 (Per-File RWMutex Map)** |
| **동시성** | 순차적 처리 (병목) | **파일 단위 동시 처리** |
| **구현 복잡도** | 낮음 (3줄) | 중간 (sync.Map + reference count) |
| **이유** | 구현이 가장 간단함 | 서버 환경의 성능 병목 방지 |

**교훈**: AI는 "구현이 간단하다"는 이유로 DB-level Lock을 추천했습니다. 하지만 사용자는 서버 환경에서의 **성능 병목**을 예측하고 더 정교한 Lock 전략을 요구했습니다. "AI가 쉽다고 하는 것이 항상 옳은 것은 아니다" — 오히려 AI의 쉬운 방법이 **사용자가 알아야 할 함정**일 수 있습니다.

## Spec 단계 — 모든 결정을 하나로 통합

모든 ADR이 승인되자, Cocrates는 모든 결정을 하나의 문서로 통합했습니다.

> `spec/jsondb.md` — 모듈 경로, API 시그니처, REST 엔드포인트 10개, CLI 명령어, 테스트 요구사항 등 모든 결정을 bullet으로 명시

Spec은 **자체 완결성(Self-Containment)** 원칙을 따랐습니다. Spec만 읽으면 ADR을 열어볼 필요가 없도록 모든 결정의 근거와 결과를 포함했습니다. 사용자가 검토하고 승인함으로써 Spec이 확정되었습니다.

## Generation 단계 — Spec만 보고 생성

Spec이 확정되자 Cocrates는 `spec-driven-generation` 스킬을 로드했습니다. 이 스킬의 핵심 원칙은 단 하나: **Spec이 유일한 생성 기준이다** — 원래 프롬프트가 아니라.

생성된 산출물:

- **Library (7개 Go 파일)**: `db.go`, `crud.go`, `path.go`, `list.go`, `schema.go`, `lock.go`, `options.go`
- **Server**: `cmd/jsondbd/main.go` — 10개 REST endpoints + graceful shutdown
- **CLI**: `cmd/jsondb/main.go` — 모든 명령어 지원 (get/set/delete/list/exists/schema/serve)
- **단위 테스트 3개**: Smoke (CRUD + list), Schema (validation + schemaless), Concurrency (goroutine safety)
- **통합 테스트**: 서버 + CLI 전체 워크플로우 검증

생성 직후 `go build`, `go vet`, 단위 테스트 3개, 통합 테스트가 실행되었고 **모두 통과**했습니다.

## Verification 단계 — 정말 Spec과 일치하는가?

생성으로 끝이 아닙니다. 다음 단계는 검증(Verification)입니다.

Cocrates는 `spec-driven-verification` 스킬을 로드했습니다. 이 스킬은 **Spec의 모든 항목을 인벤토리로 추출**하고, 생성된 산출물이 각 항목을 만족하는지 하나하나 대조합니다.

결과는 이렇게 나왔습니다.

> **총 72개 항목: 70개 pass, 1개 fail, 1개 not-verifiable**

대부분 일치했지만, **하나의 Deviation(차이)이 발견**되었습니다.

### Deviation 사례: CLI Schema URL의 이중 슬래시

Spec은 이렇게 문서화되어 있었습니다.

> `jsondb schema set /blog/episode schema.json` → `PUT /schema/blog/episode`

하지만 실제로 생성된 CLI는 이렇게 동작했습니다.

> `PUT /schema//blog/episode` (이중 슬래시)

왜 이런 일이 발생했을까요? CLI 코드를 살펴보니, URL 구성 방식이 문제였습니다.

```go
baseURL := fmt.Sprintf("%s/schema/%s", serverAddr, schemaPath)
```

`schemaPath`가 `/blog/episode` (앞에 `/`가 있는 상태)로 전달되면서 URL에 `/schema//blog/episode`라는 이중 슬래시가 생성된 것입니다.

**왜 문제인가?** 서버는 `TrimPrefix` 덕분에 정상 동작했습니다. 하지만 Spec에 문서화된 URL과 다릅니다. 다른 HTTP 클라이언트나 문서화 도구에서는 문제가 될 수 있는 미묘한 차이입니다.

**심각도**: Minor — 동작에는 문제가 없지만, Spec과의 일관성이 깨졌습니다.

**교훈**: Spec대로 생성했지만 **구현의 디테일에서 차이가 발생**할 수 있습니다. "Spec이 있으니 완벽할 것이다"라고 가정하지 말고, 반드시 검증으로 확인해야 합니다.

### 더 중요한 발견: Undocumented ASR (암묵적 설계 결정)

검증 과정에서 더 중요한 발견이 있었습니다. AI가 Spec에 없던 **6가지 설계 결정**을 스스로 내린 것입니다. Cocrates는 이것을 **Undocumented ASR(Architecture Significant Requirement)** 이라고 부릅니다.

| # | 결정 | 위치 | 위험도 | 권장 조치 |
|---|------|------|--------|-----------|
| 1 | CLI `set` 값 처리: raw string 전송, 서버에서 fallback 파싱 | CLI, Server | 낮음 | 확인 (현재 동작 합리적) |
| 2 | **URL encoding 미사용**: query parameter를 `fmt.Sprintf`로 직접 조합 | CLI | **중간** | **Spec 업데이트 필요** |
| 3 | 서버 요청 로깅 미들웨어: Spec에 없는 추가 기능 | Server | 없음 | 확인 (무해함) |
| 4 | CLI `exists` 오류 판단: 에러 메시지 문자열 heuristic | CLI | 낮음 | 확인 (현재 동작 합리적) |
| 5 | **`SetPath` 스키마 재검증 생략**: 문서 속성 변경 시 schema 검증 안 함 | Library | **중간** | **Spec 업데이트 필요** |
| 6 | `WithPort` 값 미사용: 옵션은 있지만 DB에서 실제로 읽지 않음 | Library | 없음 | 확인 (무해함) |

이 중 **#2(URL encoding)** 와 **#5(SetPath 검증)** 는 Spec에 포함되어야 할 중요한 설계 결정입니다. Spec이 이 부분을 명시하지 않았기 때문에 AI가 임의로 결정했고, 사용자가 검토할 기회를 놓쳤습니다.

**교훈**: Spec에 명시되지 않은 모든 것은 AI의 **침묵의 기본값(silent default)** 으로 채워집니다. Undocumented ASR의 발견은 단순히 "AI가 뭘 더 했나" 확인하는 것을 넘어, **Spec의 완성도를 높이는 피드백**입니다. "이 선택을 Spec에 기록해야 하는가?"라는 질문이 Spec을 더 견고하게 만듭니다.

## 회고: 세 가지 교훈

### 교훈 1 — '쉬운 방법'이라는 이름의 함정

이 실습에서 가장 중요한 발견 중 하나는 **AI가 대체로 '쉬운 방법'을 제시한다는 사실**입니다.

Cocrates가 제시한 첫 번째 접근법들은 모두 가장 구현이 간단하고, 가장 일반적인 패턴을 따르는 방향이었습니다.

| 영역 | AI의 첫 제안 (쉬운 방법) | 이유 |
|------|------------------------|------|
| 저장 모델 | Collection-Document | AI가 가장 많이 본 NoSQL 패턴 |
| 동시성 | DB-level Lock | 코드 3줄이면 끝 |
| 아키텍처 | Library-Only | 서버를 만들 필요 없음 |

하지만 사용자가 검토하고 질문할 때마다 더 나은 방향으로 수정되었습니다.

| 사용자의 질문/통찰 | 최종 결정 |
|-------------------|-----------|
| "path/filename.json으로 저장되는 것이 필요한데" | **Path-Addressable** |
| "복수 프로세스에서 요청하면 문제가 되겠네" | **Client-Server** |
| "DB 전체를 lock하는 것은 비효율인 것 같아" | **Per-File Mutex** |

### 교훈 2 — Spec만으로는 부족하다: 반복/증분 개발

ADR과 Spec으로 아키텍처를 신중히 설계했지만, **검증 단계에서 Spec과 다른 구현(Deviation)이 발견되었습니다.**

AI가 Spec대로 생성했음에도, CLI의 URL 구성 방식 같은 디테일에서 차이가 발생했습니다. 이는 **AI가 한꺼번에 많은 것을 생성할수록 실수가 증가하는 경향**과도 연결됩니다.

더 중요한 발견은 **Spec에 포함되지 않은 6가지 설계 결정(Undocumented ASR)** 이었습니다. 이 중 일부는 데이터 무결성(SetPath 검증 누락)이나 호환성(URL encoding)에 영향을 미칠 수 있는 사항이었습니다.

**이것이 의미하는 바**: "Spec을 완벽하게 만들면 끝"이 아닙니다. Spec은 살아있는 문서(living document)로서, 검증 단계에서 발견된 사항을 반영하여 **지속적으로 업데이트**되어야 합니다.

```
Verification → Spec 업데이트 → 재생성 → 재검증
```

이 **반복/증분 주기**가 구조 기반 생성의 완성된 흐름입니다.

### 교훈 3 — Undocumented ASR의 발견은 실패가 아니라 기회다

Undocumented ASR의 발견은 "AI가 잘못했다"는 의미가 아닙니다. 오히려 **Spec이 더 완벽해질 기회**입니다.

사용자가 "이런 것까지 Spec에 써야 하나?"라고 생각한 디테일이 실은 구조적 영향을 미칠 수 있는 중요한 결정이었음을 검증을 통해 깨닫게 됩니다. "Spec이 완벽하지 않아서 생긴 문제"가 아니라 "Spec이 완성되어 가는 과정"으로 이해해야 합니다.

## 정리

이 실습에서 우리는 Cocrates의 구조 기반 생성 파이프라인 전체를 경험했습니다.

1. **ADR**: AI는 '쉬운 방법'을 제시했지만, 사용자의 질문과 검토를 통해 세 가지 중요한 결정이 더 나은 방향으로 수정되었습니다.
2. **Spec**: 승인된 결정을 하나의 문서로 통합했습니다.
3. **Generation**: Spec을 기준으로 코드를 생성했습니다.
4. **Verification**: 생성된 코드가 Spec과 일치하는지 72개 항목을 검증했습니다. 1개의 Deviation과 6개의 Undocumented ASR을 발견했습니다.
5. **반복/증분**: 발견된 사항을 Spec에 반영하는 피드백 루프의 필요성을 확인했습니다.

Cocrates의 가치는 "완벽한 정답을 생성하는 것"이 아니라, **사용자가 자신의 요구사항을 더 깊이 이해하고, AI의 숨은 가정을 발견하며, Spec을 더 견고하게 다듬어나가도록 돕는 것**에 있습니다.
