# 소크라테스식 학습 활동

---

지난 편(Ep3)까지 우리는 Cocrates를 소개받고 직접 설치해보았습니다. 이제 본격적으로 Cocrates의 핵심 기능을 실습할 차례입니다.

Cocrates는 단순히 질문에 답하는 도구가 아닙니다. 사용자가 스스로 발견하고, 기록하고, 확인하도록 이끄는 학습 파트너입니다. 이 과정은 세 단계로 이루어집니다:

- **Education**: 정답을 바로 주는 대신, 질문과 미션으로 스스로 깨닫게 한다
- **Knowledge Capture**: 배운 내용을 구조화하여 지식 베이스에 기록한다
- **Reflection**: 진짜 이해했는지 다양한 각도에서 확인한다

이 세 단계가 실제로 어떻게 경험될까요? **실제 대화를 기반으로 한 실습**을 통해, Cocrates의 학습 방식을 직접 체험해보겠습니다.

---

## 실습 시나리오: Bloom's Taxonomy 배우기

우리가 배울 개념은 **블룸의 교육분류학(Bloom's Taxonomy)** 입니다. 교육학에서 가장 잘 알려진 프레임워크 중 하나로, 학습의 인지적 깊이를 여섯 단계로 나눈 것입니다.

> **기억(Remember) → 이해(Understand) → 적용(Apply) → 분석(Analyze) → 평가(Evaluate) → 창조(Create)**

혹시 이 피라미드 그림을 본 적이 있나요? 아마 대부분 있을 겁니다. 그런데 과연 우리는 이것을 **제대로** 알고 있었을까요?

Cocrates와의 대화를 따라가며 확인해봅시다.

---

## Education — 깨달음의 연속

### 첫 턴: Cocrates가 던진 질문

"블룸의 분류학에 대해 알려줘"라고 묻자, Cocrates는 예상과 다르게 행동했습니다. 개념을 길게 설명하는 대신, 아주 짧은 브리핑과 함께 **하나의 미션**을 던졌습니다.

> 어떤 교사가 학생들에게 피라미드의 낮은 단계(기억)부터 높은 단계(창조)까지 순서대로 가르치는 수업을 설계했습니다. 이 교사의 접근법에서 **한 가지 심각한 문제점**을 찾아보세요.

이 질문의 핵심은 이것입니다: ***"낮은 단계를 모두 '마스터'해야만 높은 단계로 갈 수 있다"는 가정이 과연 옳을까?***

여기서 잠시 생각해보세요. 여러분은 어떻게 답하시겠습니까?

### 첫 번째 깨달음: 피라미드는 '수업 순서'가 아니다

정답은 이것입니다. **피라미드가 위계적이라고 해서, 반드시 낮은 단계부터 순서대로 가르쳐야 하는 것은 아니다.**

블룸 분류학이 말하는 것은 **의존성 구조**입니다. 높은 단계(예: 창조)는 낮은 단계(예: 기억)를 자연스럽게 포함합니다. 진정으로 창조했다면, 그 과정에서 필요한 지식은 이미 기억하고 있다는 뜻입니다.

하지만 이것이 **"기억을 먼저 가르치고, 그다음에 창조를 가르쳐라"** 는 의미는 아닙니다.

실제로는 반대의 접근이 더 효과적인 경우가 많습니다. "프로그래밍으로 웹사이트를 만들어봐(창조)"라는 과제를 받은 학생은, 문법을 몰라서 스스로 찾아보고(기억), 왜 이렇게 동작하는지 이해하게 됩니다(이해).

> **높은 단계의 과제가 낮은 단계의 지식을 '끌어당긴다'(Pull).**

이것이 첫 번째 깨달음입니다. 평소에 우리가 당연하게 여겼던 가정 — "기초를 먼저 배워야 한다" — 이 뒤집혔습니다.

### 두 번째 깨달음: 1차원이 아니라 2차원이다

Cocrates는 여기서 멈추지 않았습니다. 다음 턴에서 **또 다른 층위**가 존재한다는 사실을 드러냈습니다.

2001년, 앤더슨과 크래스월(Anderson & Krathwohl)은 블룸의 원래 분류학을 개정하면서 **두 번째 차원**을 추가했습니다. 바로 **지식 차원(Knowledge Dimension)** 입니다.

|                | 기억 | 이해 | 적용 | 분석 | 평가 | 창조 |
|----------------|------|------|------|------|------|------|
| 사실적 지식     |      |      |      |      |      |      |
| 개념적 지식     |      |      |      |      |      |      |
| 절차적 지식     |      |      |      |      |      |      |
| 메타인지 지식   |      |      |      |      |      |      |

블룸 분류학은 1차원 피라미드가 아니라 **2차원 매트릭스**였던 것입니다.

한 가지 개념도 '아는 방식'이 다릅니다. 예를 들어 자바스크립트의 `map` 함수를 보더라도:

- **사실적 지식**: `map` 함수의 문법과 정의를 암기한다
- **개념적 지식**: `map`이 입력 함수와 배열의 관계라는 원리를 이해한다
- **절차적 지식**: `map` 코드를 직접 작성하는 방법을 안다
- **메타인지 지식**: "내가 `map`을 쓸 때 자주 `this` 바인딩을 깜빡한다"는 자기 인식을 가진다

이것들은 계층이라기보다 **층위(layer)** 에 가깝습니다. 절차적 지식이 사실적+개념적 지식을 전제하긴 하지만, 학습 순서가 반드시 사실→개념→절차일 필요는 없습니다.

> "이걸 읽기 전에는 몰랐는데, 블룸 분류학에 2번째 차원이 있는지 전혀 몰랐다" — 이것이 두 번째 깨달음입니다.

### 세 번째 깨달음: Push vs Pull — 어떻게 가르칠 것인가

대화가 진행될수록, 하나의 근본적인 질문으로 수렴했습니다.

> "낮은 단계가 선행 조건이 아니라는 원칙"과 "높은 단계는 낮은 단계를 포함한다"는 원칙 — 이 두 가지는 모순되는 것처럼 보인다. 어떻게 이해해야 할까?

Cocrates는 이 모순을 해소하기 위해 **Push와 Pull**이라는 두 가지 교수 전략을 제시했습니다.

| | Push | Pull |
|:--|:--|:--|
| **방향** | 낮은 단계 → 높은 단계로 순차적 | 높은 단계 과제를 먼저 던짐 |
| **적합한 대상** | 초보자, 인지 과부하 상태 | 선지식이 있는 학습자 |
| **장점** | 안전하게 따라올 수 있음 | 동기 부여가 강함, 필요에 의해 학습 |
| **단점** | 지루함, 동기 저하 | 인지 부하가 클 수 있음 |

예를 들어 Git 브랜치 병합을 가르친다면:

- **Push 방식**: `git merge` 옵션 암기(사실적/기억) → fast-forward vs 3-way 차이 이해(개념적/이해) → 충돌 병합 실습(절차적/적용)
- **Pull 방식**: "지금 이 저장소에서 충돌이 났습니다. 직접 해결해보세요."(절차적/적용) → 충돌 메시지를 읽으며 merge 옵션 검색(사실적/기억 — 자연 발생) → fast-forward와 3-way의 차이가 궁금해짐(개념적/이해 — 자연 발생)

**교육의 기술은 어느 한 전략만 고집하는 것이 아니라, 학습자의 상태에 따라 두 전략을 혼합하는 것**입니다.

초보자에게 Pull만 강요하면 인지 과부하로 좌절합니다. 반대로, 어느 정도 배경 지식이 있는 학습자에게 Push만 하면 지루함과 동기 저하가 옵니다. Cocrates의 education 스킬은 **기본적으로 Pull을 사용하지만, 사용자가 인지적 붕괴를 보이면 Push로 전환**하도록 설계되어 있습니다.

### 직접 설계해보는 Create 미션

Education 단계의 마지막은 **창조(Create)** 였습니다. Cocrates는 이렇게 질문했습니다.

> 지금까지 배운 내용을 바탕으로, 신입 개발자 3명을 위한 **"Git 브랜치 전략" 4시간 교육 커리큘럼**을 설계해보세요.
> 조건: 최소 4개 인지 과정 포함, Push와 Pull 전략을 각각 한 번 이상 사용, 마지막 세션은 Create로 마무리, 지식 차원도 명시.

이 미션을 수행하면서, 단순히 '아는 것'을 넘어 '설계할 수 있는 것'으로 이해가 확장되었습니다. 실제로 설계한 커리큘럼은 이후 Cocrates의 피드백을 받아 **기억→이해→적용→분석→창조**의 5개 인지 과정을 포함하고, Push와 Pull을 혼합한 완성된 교육 설계로 발전했습니다.

## Knowledge Capture — 배운 내용을 기록하다

Education 단계가 끝난 후, "배운 내용을 정리해줘"라고 요청하자 Cocrates는 `kb/bloom-taxonomy.md` 파일을 생성했습니다.

Knowledge Capture의 핵심은 **상세한 설명이 아니라, 나중에 떠올릴 수 있는 핵심만 기록하는 것**입니다. 특히 중요한 것은 **"내가 틀렸던 가정(Gap)"** 을 기록하는 것입니다.

실제로 저장된 내용 중 일부입니다:

> **내가 틀렸던 가정 / Gap**
> - 블룸 분류학이 단순한 1차원 피라미드인 줄 알았는데, 2차원 매트릭스가 원래 모델에 더 가까움
> - "낮은 단계를 먼저 가르쳐야 한다"고 생각했지만, 반드시 순차적일 필요가 없으며 Pull 전략이 더 효과적인 경우가 많음
> - 교육 커리큘럼을 설계할 때 인지 과정 단계별로 활동을 명시적으로 매핑하는 방법을 잘 몰랐음

**무지(ignorance)의 기록** — 이것이 Knowledge Capture의 핵심입니다. 내가 무엇을 몰랐는지를 기록하는 것이 가장 강력한 학습 도구입니다. 나중에 이 파일을 다시 읽으면, 단순한 요약보다 훨씬 더 생생하게 당시의 깨달음이 떠오릅니다.

## Reflection — 진짜 아는지 확인하다

마지막 단계는 Reflection입니다. "내가 제대로 이해했는지 확인해줘"라고 요청하자, Cocrates는 **interviewer**로 변신했습니다.

여기서 중요한 점은 Cocrates가 **Pull 전략**을 사용한다는 것입니다. 즉, 높은 단계의 질문을 먼저 던지고, 사용자가 얼마나 깊이 답할 수 있는지 관찰합니다. Push(순차적 확인)가 아니라 Pull(도전적 질문 → 경계 발견)로 접근하는 이유는, Reflection의 목적이 '가르치는 것'이 아니라 '진짜 아는 영역과 아닌 영역의 경계를 찾는 것'이기 때문입니다.

실제로 던져진 질문 중 하나:

> "2차원 매트릭스의 각 셀에 학습 목표를 배치해보세요. 예를 들어 'Git 브랜치 전략 교육 커리큘럼'의 각 세션을 2차원 매트릭스 어디에 위치시킬 수 있을까요?"

이 질문에 답하면서, 개념을 '설명할 수 있는 수준'과 '실제로 적용할 수 있는 수준'의 차이가 명확히 드러났습니다. 일부 셀은 자신 있게 채웠지만, 특정 조합(예: 메타인지 × 분석)에서는 막힘이 있었습니다.

Reflection의 결과는 두 가지로 정리됩니다:

- ✅ **잘 알고 있는 것** — 2차원 매트릭스의 구조, Push/Pull 전략의 차이, 피라미드의 의미
- ⚠️ **아직 불명확한 것** — 실제 교육 설계에 2차원 매트릭스를 완전히 활용하는 방법

> ⚠️ 항목이 나쁜 것이 아닙니다. 오히려 이것이 Reflection의 진정한 가치입니다. "모르는 것을 아는 것"이 학습의 출발점이기 때문입니다.

## 전체 사이클 회고

이 실습을 통해 우리는 Cocrates의 Learning 파이프라인 **3단계를 모두 경험**했습니다.

```
Education → Knowledge Capture → Reflection
```

각 단계의 특징을 정리하면:

| 단계 | 경험한 내용 | 나의 역할 |
|:--|:--|:--|
| **Education** | 블룸 분류학의 2차원 매트릭스, Push/Pull 전략을 **질문과 미션을 통해 스스로 발견** | 질문에 답하고, 미션을 수행하고, "내가 몰랐던 것"을 인식 |
| **Knowledge Capture** | 배운 내용을 `kb/bloom-taxonomy.md`에 **핵심과 Gap 중심**으로 기록 | 저장 승인, 기록된 Gap을 확인 |
| **Reflection** | 2차원 매트릭스에 학습 목표를 배치하는 **Pull 방식의 평가** | 질문에 답하며 진짜 아는 영역과 아닌 영역의 경계 확인 |

그런데 이 실습은 한 가지 더 중요한 것을 보여주었습니다.

**이 실습 자체가 Push와 Pull 전략의 실제 사례였습니다.**

Cocrates는 Bloom's Taxonomy라는 개념을 가르치기 위해 **Pull 전략**을 사용했습니다. 즉, "블룸 분류학에 대해 알려줘"라는 요청에 개념을 쭉 설명하지 않고(Push), 오히려 결함이 있는 예시와 미션을 먼저 던졌습니다(Pull). 독자인 여러분도 이 글을 읽으면서 중간중간 "잠깐, 나는 어떻게 답했을까?"라고 스스로에게 질문했을 것입니다.

그것이 Cocrates의 교육 방식입니다. 단순히 정보를 전달하는 것이 아니라, **당신이 스스로 깨닫도록 이끄는 것**.

> 이 실습에서 가장 먼저 바뀐 가정은 무엇인가요? 아마 "나는 블룸 분류학을 알고 있다"고 생각했던 것이 가장 먼저 깨진 가정일 것입니다.

이번 편에서는 Cocrates의 Learning 파이프라인을 실제 대화를 기반으로 경험해봤습니다. Education은 질문으로 발견하게 하고, Knowledge Capture는 기록하게 하며, Reflection은 확인하게 합니다. 그리고 이 모든 과정은 사용자가 **주체적으로 참여**할 때만 의미가 있습니다.
