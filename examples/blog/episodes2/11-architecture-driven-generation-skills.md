# 구조 기반 산출물 생성 활동 — 스킬

---

지난 이야기에서는 구조 기반 산출물 생성의 '왜'를 살펴보았습니다. 구조가 먼저여야 하는 이유, ASR과 ADR의 개념, 그리고 다섯 단계 파이프라인의 흐름을 이해했습니다.

이번에는 그 파이프라인을 실제로 구현하는 **네 개의 스킬**을 들여다봅니다. 각 스킬이 무엇을 담당하고, 어떤 원칙으로 작동하며, 어떻게 연결되어 있는지 — 핵심 내용을 설명합니다.

각 스킬은 고유한 원칙과 금지 사항을 가집니다. 이 경계가 파이프라인의 품질을 결정합니다.

---

## 1. spec-driven-generation — 설계도대로 짓는다

첫 번째 스킬은 **spec-driven-generation**입니다. 이름 그대로 Spec에 따라 최종 산출물을 생성합니다.

### Spec이 유일한 근거

이 스킬의 가장 중요한 원칙은 **Spec이 유일한 근거**라는 점입니다. 원래 사용자가 처음에 말한 프롬프트가 아닙니다. 중간에 오간 대화의 내용도 아닙니다. 오직 Spec에 기록된 요구사항과 결정과 제약만이 생성의 기준입니다.

프롬프트와 Spec이 충돌하면 어떻게 될까요? **Spec을 따릅니다.** 사용자가 처음에는 A라고 말했지만, 이후 ADR과 Spec 작성 과정을 거치면서 B로 결정을 바꿨다면, Spec의 B가 우선입니다. 이것이 구조 기반 생성의 핵심입니다. 충동적인 첫 요청이 아니라, 충분히 검토된 결정이 반영된 Spec이 최종 기준이 되는 것입니다.

### ASR Readiness Gate

그런데 Spec이 없는 상태에서 "생성해줘"라는 요청이 들어오면 어떻게 될까요?

여기서 **ASR Readiness Gate**가 작동합니다. Cocrates는 무작정 생성하지 않습니다. 대신 이렇게 합니다:

1. 산출물 유형을 추론한다
2. 해당 산출물에 적용되는 ASR(구조적으로 중요한 요구사항)을 식별한다
3. ASR을 사용자와 함께 검토한다: 결정됨(decided) / 암시됨(implied) / 열림(open)
4. 열린(open) ASR 중 대안이 필요한 것은 ADR을 작성한다
5. 모든 결정이 완료되면 Spec으로 통합한다
6. 그제야 생성한다

이것이 Cocrates가 다른 AI 도구와 가장 크게 다른 점입니다. 다른 도구는 "생성해줘"라는 말에 바로 결과물을 내놓습니다. Cocrates는 "생성하기 전에 먼저 결정해야 할 것들이 있습니다"라고 말하고, 그 결정 과정을 함께합니다.

### ASR의 보편 범주

ASR은 산출물 유형에 따라 다르지만, 모든 산출물에 공통으로 적용되는 8가지 범주가 있습니다:

1. **목적과 독자**: 누구를 위한 것인가, 무슨 일을 하는가
2. **산출물 형태**: 어떤 유형, 형식, 출력 위치인가
3. **범위 경계**: 포함되는 것과 제외되는 것
4. **품질 기준**: '완료'와 '충분히 좋은'의 기준
5. **제약**: 언어, 마감일, 브랜드 가이드라인, 예산
6. **구조와 구성**: 내용이나 기능이 어떻게 배치되는가
7. **통합과 의존성**: 무엇과 연결되거나 무엇을 기반으로 하는가
8. **프로세스와 단계**: 승인 게이트나 반복 전달 방식

이 범주들을 보면, 일반적인 AI 사용에서 얼마나 많은 것들이 '침묵의 기본값'으로 결정되는지 짐작할 수 있습니다. "보고서 작성해줘"라는 한 마디에 AI는 독자도, 형식도, 범위도, 품질 기준도 스스로 결정합니다. 그리고 그 결정이 당신의 의도와 맞을 확률은 얼마나 될까요?

## 2. adr-writing — 왜 이런 결정을 내렸는가

두 번째 스킬은 **adr-writing**입니다. 여기서 ADR은 **Any Decision Record**의 약자입니다. 원래 소프트웨어 업계에는 Architecture Decision Record라는 것이 있습니다. 하지만 Cocrates는 이것을 더 넓은 의미로 확장했습니다. 기술 아키텍처뿐 아니라 문서 구조, 발표 순서, 블로그 시리즈의 구성처럼 **어떤 종류의 결정이든** ADR로 기록할 수 있습니다.

### ADR의 핵심 구조

ADR의 기본 원칙은 단순합니다. **하나의 ADR 파일 = 하나의 Concern(관심사)**입니다. 층수 결정에 대한 ADR, 지붕 형태에 대한 ADR은 각각 별도의 파일입니다.

각 ADR은 이렇게 구성됩니다:

- **Concern**: 검토 중인 질문 (예: "이 보고서의 도입부를 어떻게 구성할 것인가?")
- **Options**: 최소 2~3개의 실질적 대안
- **Tradeoffs**: 대안별 비교
- **Decision**: 사용자가 승인한 결정

### Options를 다루는 방식

adr-writing 스킬이 일반 AI 사용과 가장 크게 다른 점은 **Options 제시 방식**에 있습니다.

보통 AI에게 선택지를 요청하면 장황한 설명이 이어집니다. "Option A는 이것이고 장점은... 단점은... Option B는 저것이고..." 하지만 adr-writing은 간결한 bullet을 사용합니다. 각 Option은 이름 + 2~4개의 bullet으로 요약합니다.

```
Option A — MySQL
- 관계형 데이터베이스, 성숙한 생태계
- 장점: 강력한 일관성, 익숙한 SQL 도구
- 단점: 다양한 속성을 가진 아이템에 유연성 부족

Option B — PostgreSQL
- 관계형 + JSONB으로 반정형 데이터 지원
- 장점: 필요한 곳은 관계형, 유연성이 필요한 곳은 JSONB
- 단점: 혼합 접근에 약간 복잡한 쿼리
```

이 형식이 중요한 이유는 **비교를 쉽게 만들기 때문**입니다. 장황한 설명은 오히려 결정을 어렵게 만듭니다. 핵심만 간결하게 제시해야 사용자가 쉽게 비교하고 결정할 수 있습니다.

### 승인 게이트

ADR은 `proposed` 상태로 저장됩니다. 그다음 사용자에게 검토를 요청합니다.

여기서 Cocrates는 절대로 사용자의 명시적 승인 없이 결정을 확정하지 않습니다. "이 Option이 가장 좋은 것 같습니다. 승인하시겠습니까?"라고 묻고, 사용자가 "네, 그걸로 하죠"라고 말한 후에야 `approved`로 변경합니다. 그리고 Spec 작성 단계로 넘어갑니다.

### 파일로 저장된다

ADR은 프로젝트 루트의 `adr/` 디렉토리에 `{concern-slug}.md` 형식으로 저장됩니다. 이 파일들은 단순한 기록이 아니라 **의사결정 감사 추적(audit trail)** 입니다. 나중에 "왜 이렇게 결정했더라?" 하고 궁금해질 때, ADR 파일을 열어보면 선택지와 근거가 생생하게 남아 있습니다.

## 3. spec-writing — 결정을 하나의 설계도로

세 번째 스킬은 **spec-writing**입니다. ADR에서 승인된 결정과 사용자의 요구사항을 **Spec(명세서)**으로 통합합니다.

### Spec의 핵심 원칙: 자체 완결성

Spec의 가장 중요한 원칙은 **자체 완결성(Self-Containment)**입니다. 즉, Spec 하나만 읽으면 '무엇을 만들지'를 완전히 이해할 수 있어야 합니다.

이것이 의미하는 바는 이것입니다. Spec을 읽는 사람(또는 AI)이 ADR 파일을 따로 열어볼 필요가 없어야 합니다. ADR에는 선택지와 비교 분석이 들어 있습니다. Spec에는 그중 승인된 결정만 '명확한 요구사항'의 형태로 기록됩니다.

예를 들어, ADR에는 이렇게 기록되어 있습니다:

> Option B — PostgreSQL 선택. 관계형 무결성 + JSONB 유연성.

그러면 Spec에는 이렇게 기록됩니다:

> - 저장소: PostgreSQL. 핵심 필드는 관계형, 가변 속성은 JSONB 컬럼 사용

차이가 보이시나요? Spec은 '무엇을 할지'만 명확히 적고, '왜 그렇게 결정했는지'는 기록하지 않습니다. '왜'는 ADR이 담당합니다.

### ADR vs Spec

| | ADR | Spec |
|---|-----|------|
| 단위 | 하나의 Concern | 하나의 산출물 또는 요구사항 범위 |
| 목적 | 대안 비교, 결정 근거 기록 | 결정된 사항을 명확히 명시 |
| 독자에게 필요한 것 | 선택지와 장단점 이해 | 무엇을 만들지 이해 |
| 파이프라인 역할 | 의사결정 감사 추적 | 생성과 검증의 유일한 입력 |

### Spec은 살아있는 문서

Spec은 한 번 쓰고 끝나는 문서가 아닙니다. 사용자가 직접 편집할 수 있고, 요구사항이 바뀌면 업데이트됩니다. 그래서 Spec에는 `Status`나 `Approved` 같은 승인 상태 필드가 없습니다. 승인 여부를 추적하는 것이 아니라, **현재 합의된 요구사항이 무엇인지**를 기록하는 문서이기 때문입니다.

Spec의 서술 방식도 중요합니다. 장황한 산문 대신 간결한 bullet을 사용합니다. 하나의 bullet = 하나의 요구사항, 결정, 또는 제약입니다. 그리고 각 요구사항은 검증 가능한(verifiable) 문장으로 작성합니다. 나중에 "이 요구사항이 충족되었는가?"라고 물었을 때 명확히 판단할 수 있어야 하기 때문입니다.

## 4. spec-driven-verification — 설계도와 같은지 확인한다

마지막 스킬은 **spec-driven-verification**입니다. 생성된 산출물이 Spec과 일치하는지 확인합니다.

### 검증의 이중 목적

이 스킬의 검증은 두 가지 목적을 가집니다.

**첫째, 결과물이 Spec과 일치하는지 항목별로 확인합니다.** Spec의 모든 요구사항이 충족되었는지, 제약을 위반한 것은 없는지, 범위를 벗어난 내용이 포함되지는 않았는지 하나하나 검증합니다.

**둘째, Undocumented ASR(문서화되지 않은 구조적 요구사항)을 발견합니다.** 이것이 더 흥미로운 부분입니다. 생성 과정에서 AI가 Spec에 없던 설계 결정을 결과물에 포함시켰다면, 그것을 발견하는 것입니다.

예를 들어, "분기별 매출 보고서를 작성해줘"라고 Spec에 적었습니다. AI가 보고서를 만들었는데, 섹션 순서를 '요약 → 지역별 매출 → 제품별 매출 → 전년 대비 비교 → 전망'으로 구성했습니다. 그런데 이 섹션 순서는 Spec에 없던 결정입니다. AI가 임의로 선택한 것입니다. 이 섹션 순서는 결과물의 품질에 영향을 미치는 구조적 결정이므로, **Undocumented ASR**이 됩니다.

### Deviation과 Undocumented ASR의 차이

| | Deviation | Undocumented ASR |
|---|-----------|------------------|
| 관계 | Spec 항목을 위반함 | Spec에 없었지만 구현됨 |
| 예시 | "PostgreSQL 사용"이라고 했는데 MongoDB로 만듦 | 섹션 순서를 AI가 임의로 결정함 |
| 조치 | 수정이 원칙 | 확인(confirm), Spec 업데이트, 또는 수정 중 선택 |

중요한 것은 Undocumented ASR이 '잘못'은 아니라는 점입니다. AI의 선택이 오히려 더 합리적일 수도 있습니다. 문제는 그 결정이 **검토되지 않았다**는 점입니다. 이 스킬의 역할은 그 결정을 발견해서 사용자에게 보여주고, "이 결정이 괜찮은가요?"라고 묻는 것입니다.

### 검증 결과는 파일로 저장된다

검증 결과는 프로젝트 루트의 `verification/` 디렉토리에 저장됩니다. Spec 항목별 상태(pass/fail/partial)와 증거, Deviation 목록, Undocumented ASR 목록이 기록됩니다.

사용자는 이 파일을 검토한 뒤, 각 항목에 대해 어떻게 처리할지 결정합니다.

## 구조 기반 산출물 생성 활동의 확장

지금까지 설명한 ADR → Spec → Generation → Verification 파이프라인은 기본 구조입니다. 산출물 유형에 따라 변형될 수 있습니다. 예를 들어, 이 블로그 시리즈를 만들 때는 blog-series-authoring 스킬이 overview → outline → episodes.md → episode 본문 순서로 점진적으로 구체화 합니다.
