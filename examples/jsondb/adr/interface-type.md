# CLI vs 서버 인터페이스 비교

## Concern
jsondb의 인터페이스로 CLI와 서버 중 어떤 것이 적합한가?

## Status
approved

## Context
- 여러 에이전트(프로세스)가 동시에 jsondb에 접근할 수 있음
- 필드 레벨 접근 효율성이 요구됨
- 동시성 제어가 필요
- Go 언어로 구현 (단일 바이너리 배포 가능)

## Decision
**하이브리드 방식 (CLI + Server)** 사용자 승인: CLI는 Server에 HTTP 요청을 전달하고, Server는 jsondb 엔진 모듈과 HTTP 인터페이스 서버 모듈로 구분.

## Options

### Option A — CLI (Command-Line Interface)

**동작 방식**:
```bash
# 에이전트가 subprocess로 호출
$ jsondb get data.json overview.title
"기존 제목"

$ jsondb set data.json overview.title "새 제목"
$ jsondb query data.json ".tags[?(@=='important')]"
```

**장점**:
- **단순성**: 파일을 인자로 전달하면 됨, 상태 관리 불필요
- **테스트 용이성**: 터미널에서 즉시 테스트 가능
- **배포 간이성**: 단일 바이너리, 설치 불필요
- **유연성**: 모든 언어에서 subprocess로 호출 가능
- **디버깅 용이성**: 출력을 직접 확인 가능
- **독립성**: 서버 프로세스 없이 동작

**단점**:
- **프로세스 오버헤드**: 호출마다 새 프로세스 생성 (수 ms)
- **상태 비유지**: 매번 파일을 새로 열어야 함
- **동시성 제한**: 파일 잠금으로만 제어 가능
- **빈번한 호출 시 비효율**: 초당 수십 번 호출 시 병목
- **파일 접근**: 매번 전체 파일 열기/닫기 필요

**성능 특성**:
```
단일 호출: ~5-10ms (프로세스 생성 + 파일 I/O)
배치 처리: 파일당 1회 I/O로 여러 작업 가능
동시성: 파일 레벨 잠금 (flock)
```

**적합한 경우**:
- 호출 빈도가 낮음 (하루 수십 번)
- 에이전트가 독립적으로 동작
- 간단한 CRUD 작업
- 빠른 프로토타이핑

---

### Option B — Server (IPC/네트워크 서버)

**동작 방식**:
```bash
# 서버 시작 (백그라운드)
$ jsondb serve --socket /tmp/jsondb.sock
# 또는
$ jsondb serve --port 8080

# 에이전트가 클라이언트로 접근
$ jsondb-cli get overview.title
$ jsondb-cli set overview.title "새 제목"
```

**장점**:
- **동시성 자연 해결**: 서버가 단일 파일 관리
- **상태 유지**: 파일을 메모리에 캐싱 가능
- **높은 성능**: 프로세스 오버헤드 없음, 연결 재사용
- **효율적 I/O**: 메모리 매핑 + 캐싱으로 필드 접근
- **트랜잭션 지원**: 여러 작업을 원자적으로 처리
- **동시 읽기/쓰기**: 서버가 스케줄링

**단점**:
- **복잡성**: 서버 프로세스 관리 필요
- **배포 복잡**: 서버 시작/중지 관리
- **의존성**: 서버가 죽으면 모든 클라이언트 영향
- **보안**: 소켓/포트 관리 필요
- **디버깅**: 서버 로그 분석 필요

**성능 특성**:
```
단일 호출: ~0.5-2ms (IPC 오버헤드만)
배치 처리: 여러 작업을 하나의 연결로 묶기 가능
동시성: 서버 내부에서 처리 (락, 큐)
```

**적합한 경우**:
- 호출 빈도가 높음 (초당 수 번 이상)
- 여러 에이전트가 동시 접근
- 장기 실행 애플리케이션
- 고성능 요구

## Tradeoffs

| 기준 | CLI | Server |
|------|-----|--------|
| **구현 복잡도** | ★★★★★ (매우 단순) | ★★☆☆☆ (복잡) |
| **단일 호출 성능** | ★★★☆☆ (5-10ms) | ★★★★★ (0.5-2ms) |
| **동시성 처리** | ★★☆☆☆ (파일 잠금) | ★★★★★ (서버 내부 관리) |
| **배포 용이성** | ★★★★★ (단일 바이너리) | ★★★☆☆ (서버 관리 필요) |
| **테스트 용이성** | ★★★★★ (터미널 테스트) | ★★★☆☆ (서버 시작 필요) |
| **상태 관리** | ★★☆☆☆ (상태 없음) | ★★★★★ (캐싱 가능) |
| **장기적 확장성** | ★★☆☆☆ (제한적) | ★★★★★ (유연) |
| **에이전트 통합** | ★★★★☆ (subprocess) | ★★★☆☆ (IPC/네트워크) |

## 하이브리드 방식 (Option C) — 추천

**CLI + Server 조합**:

```bash
# 독립 실행 시 CLI 모드
$ jsondb get data.json title

# 서버 모드로 실행 시
$ jsondb serve --socket /tmp/jsondb.sock

# 클라이언트 모드로 접근
$ jsondb --socket /tmp/jsondb.sock get title
```

**장점**:
- CLI의 단순성 + Server의 성능/동시성
- 사용자가 선택 가능
- 점진적 마이그레이션 가능

**단점**:
- 구현량 증가
- 두 가지 모드 관리 필요

## Recommendation

**하이브리드 방식 (Option C)** 추천 이유:

1. **유연성**: CLI로 빠른 테스트, Server로 프로덕션 사용
2. **점진적 도입**: 먼저 CLI로 시작, 필요 시 Server 모드 추가
3. **에이전트 호환**: 어떤 에이전트든 접근 가능 (subprocess 또는 IPC)
4. **Go의 장점 활용**: 단일 바이너리로 두 모드 모두 제공 가능

## 구체적 아키텍처

사용자가 결정한 아키텍처 구조:

```
┌─────────────┐     HTTP 요청      ┌─────────────────────────────────────┐
│             │ ─────────────────▶ │         Server 모듈                 │
│     CLI     │                    │  ┌─────────────┐  ┌──────────────┐ │
│  (Client)   │ ◀───────────────── │  │ HTTP 인터페이스│  │ jsondb 엔진 │ │
│             │     HTTP 응답      │  │   서버 모듈   │  │    모듈     │ │
└─────────────┘                    │  └─────────────┘  └──────────────┘ │
                                   └─────────────────────────────────────┘
```

- **CLI**: Server에 HTTP 요청을 전달하는 클라이언트
- **Server 모듈**: 
  - HTTP 인터페이스 서버 모듈: HTTP 요청 처리
  - jsondb 엔진 모듈: 핵심 데이터베이스 로직

## Downstream Concerns
