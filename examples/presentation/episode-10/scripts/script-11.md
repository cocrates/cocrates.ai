# Script: Slide 11 — 4️⃣ 4단계 — Generation & Verification

**Time:** ~1분

**[SLIDE 11: Two-column layout — 좌: Generation (Spec 기준 생성) / 우: Verification (✅ Spec 충족 + ❌ Undocumented ASR) / 하단: 순환 화살표 (Generation → Verification → Spec 개정 → Re-Generation)]**

(마지막 단계로 넘어가며)

4단계는 **Generation & Verification**. 생성과 검증입니다.

(화면을 가리키며)

여기서 한 가지 핵심 원칙이 있습니다. **생성의 기준은 최초 프롬프트가 아니라, 3단계에서 승인된 Spec입니다.**

(비유로 설명)

건축 비유로 보면, 설계도를 시공팀에 넘기고 시공하는 단계입니다. 그리고 완공되면 설계도와 대조하며 검수합니다. 설계도에 옥탑방이라고 쓰여 있는데 그냥 옥상으로 끝나버렸다면? 바로 문제입니다.

(두 가지를 구분하며)

검증은 두 가지를 확인합니다.

✅ **Spec 요구사항 충족 여부.** Spec에 명시된 모든 요구사항이 생성물에 반영되었는가?

❌ **Undocumented ASR.** 설계도에 없는 AI의 임의적 결정이 밀수입되지는 않았는가? 이것이 더 위험합니다. AI가 "아, 이쯤이면 사용자가 원하겠지"하고 자체적으로 결정한 내용이 결과물에 숨어 있을 수 있습니다.

(철학적으로)

**💡 당신이 검토한 결정만 가치가 있습니다.** 에피소드 2에서 다룬 **"The Unexamined Code Is Not Worth Generating"** — 검토되지 않은 산출물은 생성할 가치가 없다는 원칙이 바로 여기서 구현됩니다.

(순환을 강조하며)

**💡 검증에서 발견된 Undocumented ASR은 다음 Spec 개정의 재료가 됩니다.** 즉, 이 파이프라인은 **순환적으로 개선**됩니다. 한 번 생성하고 끝이 아니라, 검증 결과를 토대로 Spec을 개선하고 다시 생성하는 선순환 구조입니다.

(화면 전환)
