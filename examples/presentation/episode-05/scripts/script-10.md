# Script: Slide 10 — 🔍 Verification — 설계 검증 + 동시에 설계 재검토

**Time:** ~2분

**[SLIDE 10: Verification — 설계 검증 + 동시에 설계 재검토]**

**역할 1: "사양대로 구현되었는가?"**
- Spec에서 **72개 체크 항목** 식별
- 각 항목이 설계(Spec)대로 구현되었는지 확인
- 결과: ✅ 71개 PASS, ❌ 1개 FAIL (Deviation)
- 발견된 Deviation: Spec은 `PUT /schema/blog/episode` → 코드는 `PUT /schema//blog/episode` (이중 슬래시)

**복잡도가 높을수록 AI가 사양을 놓칠 가능성은 높아진다**
→ **작은 단위로 점진적/반복적 개발이 필요하다**

**역할 2: "설계 자체가 올바른가?"**
- 검증하면서 동시에 설계의 올바름을 재검토
- "사양대로 안 되면 AI 잘못"이 아니라 "설계 자체를 변경해야 하는 상황"일 수 있다

**💡 포인트:** Verification은 Spec 확인 + 설계 재검토의 기회다

---

코드가 생성되었습니다. 하지만 여기서 끝이 아닙니다. 이제 **검증** 단계가 시작됩니다.

Verification의 첫 번째 역할은 **"사양대로 구현되었는가?"** 를 확인하는 것입니다.

어떻게 확인할까요? Spec에서 **72개의 체크 항목**을 식별합니다. 그리고 각 항목에 대해 설계, 즉 Spec에 맞춰 구현되었는지를 하나하나 확인합니다.

결과는 어땠을까요? 71개는 통과, 1개는 실패. 바로 **Deviation**을 발견한 겁니다.

무엇이 문제였을까요? Spec에는 `PUT /schema/blog/episode`라고 명시되어 있었는데, 실제 코드는 `PUT /schema//blog/episode` — 슬래시가 하나 더 들어가 있었습니다. 이중 슬래시였죠.

서버가 유연하게 처리해서 동작 자체는 문제없었지만, **API 규격의 일관성을 위반**한 겁니다. 검증이 없었다면 절대 발견되지 않았을 결함입니다.

(잠시 멈춤)

여기서 한 가지 중요한 점을 말씀드리고 싶습니다.

**소프트웨어의 복잡도가 높을수록 AI가 사양을 놓칠 가능성은 높아집니다.** jsondb는 단순했지만, 실제 소프트웨어는 훨씬 복잡합니다. 사양이 방대할수록 AI가 모든 세부사항을 정확히 구현할 확률은 낮아집니다.

그렇다면 어떻게 해야 할까요? **작은 단위로 점진적, 반복적으로 개발**하는 것이 좋습니다. 큰 Spec 한 번에 구현하고 검증하려면 부담이 너무 큽니다. 작은 Spec씩 구현하고, 검증하고, 피드백을 받고, 다음 Spec으로 넘어가는 선순환이 필요합니다.

그리고 Verification의 **두 번째 역할**이 있습니다. 바로 **"설계 자체가 올바른가?"** 를 재검토하는 것입니다.

소프트웨어 복잡도가 높은 경우, 설계 시점에 충분히 고려하지 못해서 부적절한 결정을 내리는 경우가 있습니다. 이때 "사양대로 구현되지 못했다"는 사실이 **설계의 문제를 드러내는 신호**가 될 수 있습니다.

다시 말해, 사양대로 안 나왔을 때 무조건 "AI가 잘못했다"고 생각할 게 아니라, **"설계 자체를 변경해야 하는 상황"** 일 수도 있다는 겁니다.

(잠시 멈춤)

Verification은 단순한 체크리스트가 아닙니다. Spec대로 구현되었는지 확인하는 동시에, 설계 자체가 올바른지 재검토하는 소중한 기회입니다.
