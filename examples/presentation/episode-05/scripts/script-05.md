# Script: Slide 05 — ADR ② — 아키텍처 논쟁

**Time:** ~1.5분

**[SLIDE 05: ADR ② 아키텍처 논쟁]**

**상황:** "시스템 구조는 어떻게 할까?"
**AI의 제안:** "단일 프로세스 Library-Only가 쉽고 간단합니다"
→ **핵심 문제: 쉽고 간단하다 (Simple)**
**사용자의 브레이크:** "여러 AI 에이전트가 동시에 접근할 거야. Client-Server 구조로 가야 해."
**결과:** REST API 서버(jsondbd) + CLI 도구 포함 아키텍처로 전면 수정
**💡 교훈:** 쉽고 간단하다 ≠ 올바른 설계다.

---

두 번째 논쟁입니다. 시스템을 어떻게 구성할까?

AI의 제안은 이렇습니다. "단일 프로세스 Library-Only가 쉽고 간단합니다."

AI 입장에서 가장 만들기 쉬운 방식이죠. 서버도 필요 없고, 네트워크 코드도 필요 없습니다. 그냥 라이브러리 하나 덜렁 만들어주면 끝입니다.

하지만 여기서 중요한 건, **"만들기 쉽다"는 것과 "사용하기 좋다"는 것이 완전히 다른 개념**이라는 겁니다.

사용자가 다시 한 번 브레이크를 밟습니다.

"잠깐만요. 제 환경을 생각해보세요. 여러 AI 에이전트 프로세스가 동시에 이 jsondb에 접근해야 합니다. 단일 프로세스 Library-Only로는 안 돼요. Client-Server 구조로 가야 해요."

이 한마디에 아키텍처가 완전히 바뀝니다. REST API 서버(jsondbd)와 CLI 도구를 포함한 대규모 아키텍처로 전면 수정된 거죠.

(잠시 멈춤)

여기서 얻는 교훈은 이것입니다. AI가 "쉽고 간단하다"고 말할 때, 그것은 **AI 입장에서의 쉬움**이지, **사용자의 미래를 고려한 올바른 설계**가 아닐 수 있습니다. 사용자는 현재뿐 아니라 미래의 확장성까지 고려합니다.
